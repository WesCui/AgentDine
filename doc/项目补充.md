- `LongAdder` 现在也已经在项目里**真实落地使用**了，主要用于热点缓存命中/未命中统计。

也就是说，这一块最稳妥的讲法是：

## 2. ConcurrentHashMap 在项目中的真实使用

### 2.1 落地位置

项目中 `ConcurrentHashMap` 的真实使用位置在：

- `sky-server/src/main/java/com/sky/websocket/WebSocketServer.java`

核心代码是：

```java
private static final Map<String, Session> SESSION_MAP = new ConcurrentHashMap<>();
```

这里的 `SESSION_MAP` 用来保存 WebSocket 会话，也就是“当前哪些客户端和服务端保持了长连接”。

---

### 2.2 为什么这里必须用 ConcurrentHashMap

因为 `WebSocket` 会话管理本身就是一个典型的并发场景，不是单线程逻辑。

在这个项目里，会同时发生下面几类操作：

1. 有新客户端连接进来，对应 `onOpen()`，会执行 `put`
2. 有客户端断开连接，对应 `onClose()`，会执行 `remove`
3. 支付成功后，订单服务会触发“来单提醒”，服务端要遍历所有连接并广播消息
4. 用户催单后，服务端也要再次遍历连接并广播消息

也就是说，连接建立、连接断开、消息广播这三类动作可能由不同线程同时触发。

如果这里使用普通的 `HashMap`，会有几个风险：

- 多线程并发 `put/remove/get` 时会有线程安全问题
- 广播遍历时如果同时有连接变更，容易出现并发修改问题
- 极端情况下可能导致数据不一致，甚至丢连接、重复连接、空指针等异常

而 `ConcurrentHashMap` 的价值就在于：

- 支持高并发读写
- 相比整张表一把大锁的方案，锁粒度更细
- 更适合“频繁读、也有并发写”的连接管理场景

所以这里不是“为了用并发类而用并发类”，而是因为 WebSocket 会话容器天然就应该是线程安全的。

---

### 2.3 它在业务链路里是怎么工作的

在项目里，`ConcurrentHashMap` 实际承接的是“商户端实时通知”的连接状态管理。

完整链路可以这样理解：

1. 商户端页面建立 WebSocket 连接
2. `WebSocketServer.onOpen()` 被调用
3. 当前连接对象 `Session` 以 `sid -> session` 的形式放进 `SESSION_MAP`
4. 当支付回调成功后，`OrderServiceImpl` 会调用异步通知服务
5. 异步通知服务最终调用 `webSocketServer.sendToAllClient(...)`
6. `sendToAllClient()` 会遍历 `SESSION_MAP`，向所有有效连接发送消息
7. 如果连接已经失效或关闭，则在遍历时清理掉

对应到项目，就是：

- 来单提醒
- 催单通知

这两个场景都依赖这个并发安全的连接容器。

---

### 2.4 为什么它和异步线程池是配套的

项目里不仅有 `ConcurrentHashMap`，还有异步通知线程池：

- `sky-server/src/main/java/com/sky/config/AsyncConfiguration.java`
- `sky-server/src/main/java/com/sky/websocket/WebSocketNotificationService.java`

这里的逻辑是：

- 订单主链路先完成核心状态更新
- WebSocket 推送从主链路中拆出来
- 交给独立线程池异步执行

这样一来，通知线程在广播消息时，会去访问 `SESSION_MAP`

此时如果又有新的连接进来，或者旧连接断开，连接容器仍然要保持安全可用。

所以 `ConcurrentHashMap` 和异步线程池其实是一个组合：

- 异步线程池解决“通知不要阻塞交易主流程”
- `ConcurrentHashMap` 解决“异步通知和连接变更并发发生时，会话容器仍然安全”

这就是为什么我在面试里会把这两个点一起讲。

## 3. AtomicInteger 和 LongAdder 在这个项目里应该怎么用

### 3.1 先说明真实情况

当前项目代码里：

- `LongAdder` 已经直接落地了
- `AtomicInteger` 还没有直接写进去

`LongAdder` 的真实落点是缓存指标统计。  
我现在在项目里专门做了一层轻量级缓存指标服务，核心写法是：

```java
private final Map<String, LongAdder> counters = new ConcurrentHashMap<>();
```

和：

```java
counters.computeIfAbsent(key, unused -> new LongAdder()).increment();
```

这套逻辑现在已经用于统计：

- 热点菜单查询缓存命中率
- 套餐缓存命中/未命中
- 店铺状态缓存命中/未命中

并且通过后台接口暴露成了可查看的缓存指标数据。

所以现在更准确的说法应该是：

- `LongAdder`：项目里已经真实使用了，用于高频缓存指标统计
- `AtomicInteger`：项目里还没直接写，但我能明确说出它在这个项目里适合落在哪些位置

也就是说，如果面试官问：

> “你项目里这些性能指标和并发指标，如果在系统内部做轻量级统计，会怎么落？”

我现在已经不只是“会怎么落”，而是可以直接回答：

> 我项目里已经用 `LongAdder` + `ConcurrentHashMap` 做了一版轻量级缓存指标统计，承接菜品、套餐、店铺状态这些热点缓存的命中和未命中计数；而 `AtomicInteger` 更适合我后续继续补在线连接数、重试次数、限次控制这类需要精确状态更新的场景。

---

### 3.2 LongAdder 更适合什么场景

在这个项目里，`LongAdder` 最适合用在**高并发下的全局计数指标**，而且我现在已经这么用了。

真实落地的位置是缓存指标服务，思路是：

- 用 `ConcurrentHashMap<String, LongAdder>` 维护不同缓存维度的计数器
- 命中时累加 `xxx:hit`
- 未命中时累加 `xxx:miss`
- 后台再把这些计数聚合成命中率、请求数等指标返回

这些计数有一个共同特点：

- 更新非常频繁
- 读取相对少
- 重点关注“总量统计”

这类场景下如果用 `AtomicInteger` 或 `AtomicLong`，在高并发自增时会有明显 CAS 竞争。

而 `LongAdder` 的思路是把一个热点计数拆成多个槽位分散累加，最后在读取时再汇总，所以在高并发写入下吞吐会更好。

我当前项目里最典型的落点就是缓存统计：

```java
private final Map<String, LongAdder> counters = new ConcurrentHashMap<>();
```

查询热点数据时：

- Redis 命中就给 `dish:hit`、`setmeal:hit`、`shop:hit` 这类 key 做 `increment()`
- Redis 未命中就给 `dish:miss`、`setmeal:miss`、`shop:miss` 做 `increment()`
- 后台读取时再统一汇总

最后就可以算出：

- 命中率 = `hit / (hit + miss)`
- 总请求数 = `hit + miss`

这和我简历里写的缓存治理、命中率、回源率这些说法就是能对应上的。  
并且现在项目里还有一个专门的缓存指标接口，所以面试时我可以更落地地讲：

> 我不是只在简历里写了“缓存命中率”，而是已经在代码里用 `LongAdder` 做了轻量级计数，把热点缓存访问沉淀成了可观测指标。

所以如果面试官问：

> “这些指标如果在代码里做轻量统计，你会怎么设计？”

我会回答：

> 对于高频更新、低频读取的全局统计，我优先选 `LongAdder`。我当前项目里已经在缓存指标统计里这么做了，用 `ConcurrentHashMap + LongAdder` 记录不同缓存维度的 hit/miss，再统一聚合成命中率和请求数。它比 `AtomicInteger` 更适合这种高频累加场景。

---

### 3.3 AtomicInteger 更适合什么场景

`AtomicInteger` 更适合**低到中等并发下的精确状态计数，或者需要基于 CAS 做条件更新的场景**。  
这部分虽然我当前代码里还没直接写进去，但它在这个项目里的落点也很明确。

在这个项目里，我会优先把它放到下面几类地方：

1. 在线连接数
2. 单订单通知重试次数
3. 单请求工具调用次数限制
4. 某个任务的执行状态计数

比如 WebSocket 在线连接数，如果我想在 `onOpen()` 和 `onClose()` 时维护一个实时在线商户端数量，就可以这样：

```java
private static final AtomicInteger ONLINE_COUNT = new AtomicInteger(0);
```

连接建立：

```java
ONLINE_COUNT.incrementAndGet();
```

连接关闭：

```java
ONLINE_COUNT.decrementAndGet();
```

为什么这里不一定优先用 `LongAdder`？

因为这个场景里我可能会频繁读取“当前在线人数”这个精确值，而且更新模式是加一减一，业务上更强调一个明确的、即时的数字，这时候 `AtomicInteger` 更自然。

再比如通知重试次数：

如果未来我要给某个订单提醒加“最多重试 3 次”的保护逻辑，就很适合用：

```java
private final AtomicInteger retryCount = new AtomicInteger(0);
```

然后通过：

```java
if (retryCount.incrementAndGet() > 3) {
    // 放弃重试或进入补偿逻辑
}
```

这个就体现了 `AtomicInteger` 的另一个价值：

- 不只是计数
- 还能基于 CAS 做安全的并发条件更新

所以在项目里我会这样区分：

- `LongAdder`：偏全局指标统计
- `AtomicInteger`：偏精确状态计数和限次控制

## 7. 流式输出里，SSE 和 WebSocket 怎么选？各自优缺点是什么？跨端兼容和高并发怎么考虑？

### 5 分钟答法

如果让我从项目和生产设计两个角度来回答，我会先说结论：**大模型文本生成的流式输出场景，默认我会优先选 SSE；只有在明确需要双向实时交互、前后端都要持续主动发消息时，才优先考虑 WebSocket。**

原因不是“哪个更高级”，而是通信模型不同。  
SSE 本质上还是基于 HTTP 的单向持续推送，服务端不断往客户端推送 token 增量、阶段事件和结束标记。这个模型和大模型生成天然匹配，因为大多数情况下客户端只是发起一次请求，然后等服务端持续吐 token。它的优势很明显：

1. 和现有 HTTP 网关、鉴权、日志、限流体系兼容性好  
2. 前端接入成本低，Web 端天然友好  
3. 浏览器、Node、中间层代理对它支持成熟  
4. 服务端实现简单，尤其适合“问一次、流一次、结束就断”的推理场景

SSE 的短板也有：

1. 本质是单向流，不适合复杂双向交互  
2. 在小程序、部分移动端或老旧网络环境下兼容性不一定统一  
3. 大量长连接时，本质上仍然占用连接资源，对网关和连接管理有要求

WebSocket 更适合什么？  
更适合前后端都是长生命周期连接，并且双方都会主动发消息的场景，比如：

- 多人协同编辑
- 游戏状态同步
- 客服坐席与 Agent 双向协同
- 需要在生成过程中频繁插入客户端控制信号的场景

WebSocket 的优势是双向全双工，消息格式灵活，单连接可以承接更多交互动作；但它的成本也更高：

1. 接入层复杂度更高  
2. 和现有 HTTP 鉴权、日志、监控体系天然割裂更多  
3. 在网关、负载均衡、跨端 SDK 适配上比 SSE 更重

所以回到你的项目，如果是“用户在点餐对话框里输入问题，模型逐字返回推荐结果”，这类典型单次问答式流式输出，我会优先用 SSE。因为这个场景是标准的“客户端发一次请求 -> 服务端持续输出 -> 完成后结束”，SSE 的通信模型最自然。  
如果把项目进一步扩展成“客服 Agent + 人工坐席协同 + 工具中途打断 + 前端插入确认信号”，这时 WebSocket 才会更合适。

跨端兼容上，我通常这样处理：

- `PC Web`：优先 SSE
- `App`：客户端 SDK 如果对 SSE 支持稳定，也优先 SSE；否则封一层统一流式协议
- `小程序`：很多时候原生 SSE 支持不如浏览器稳定，这时可以用分片轮询、Fetch Stream、或者统一网关把 SSE 转成客户端更易消费的流格式

高并发下的考虑不是“协议本身抗不抗压”，而是：

1. 连接生命周期多长  
2. 服务端是否能及时释放连接  
3. 是否有心跳、超时、主动断连机制  
4. 代理层和应用层能否统一治理这些长连接

所以我一般不会把这个题答成简单的“SSE 简单、WebSocket 强大”，而是会答成：**大模型流式输出默认优先 SSE，因为单向 token 流与 HTTP 生态兼容最好；WebSocket 更适合双向高交互 Agent 场景。**

### 相关知识点补充

- SSE 更像“单向事件流”
- WebSocket 更像“持续双向会话”
- 大模型流式输出大多是“服务端单向吐 token”，所以 SSE 自然
- 真正影响系统复杂度的是连接治理、超时释放、跨端兼容，而不只是协议名

### 高频追问

`Q：高并发下为什么很多团队仍然偏向 SSE？`

**回答：**

因为它和现有 HTTP 基础设施兼容度最高，鉴权、日志、限流、追踪都更容易复用，运维成本更低。

`Q：如果一个页面既要流式输出，又要中途发控制信号怎么办？`

**回答：**

一种做法是主输出走 SSE，控制动作走普通 HTTP；如果控制交互非常频繁，再切到 WebSocket 会更合理。

## 8. 流式返回时如果网络中断、前端重连、用户点击停止、或者中间要插入工具事件，该怎么设计？

### 5 分钟答法

这类题其实都围绕一个核心：**流式输出不能只看“怎么把 token 往前端吐”，而要把它当成一个有状态、有生命周期的流任务。**

我通常会把一次流式生成抽象成一个 `stream_task`，它至少要有这些状态：

- `request_id`
- `conversation_id`
- `turn_id`
- `status`，比如 `running / stopped / completed / failed`
- `last_seq`
- `partial_output`
- `usage`
- `created_at / updated_at`

有了这个抽象之后，几个问题就都好回答了。

第一，**网络中断和前端重连怎么恢复？**  
后端要给每个流片段一个递增 `seq`，同时把已经成功输出的内容按段缓存起来，最简单可以按 `request_id + seq` 存在 Redis 或内存队列里。前端重连时带上：

- 原来的 `request_id`
- 已确认收到的最后一个 `seq`

后端就可以从 `last_seq + 1` 开始继续补发，或者在模型已经结束时直接把未消费完的尾部内容一次性补回。  
这样做可以避免两类问题：

- 重复输出：因为前端和后端都按 `seq` 去重
- 丢包：因为服务端保留了短周期输出缓冲

第二，**用户点击停止生成怎么办？**  
不能只是前端把连接断掉，后端什么都不做。真正稳妥的方式是：

1. 前端发一个 `cancel` 请求或控制消息
2. 后端把 `stream_task.status` 改为 `stopped`
3. 推理线程或上游模型代理在每个 chunk 输出前检查取消标记
4. 一旦发现取消，就中断后续推理、关闭上游连接、释放缓冲和上下文对象

如果是自建推理服务，还要考虑 GPU 推理请求的取消；如果是第三方模型 API，至少要做到“停止继续消费和转发”，并在上层尽快释放会话资源。  
核心点是：**停止生成一定要有服务端状态位和资源回收逻辑，不能只靠客户端断连。**

第三，**流式返回中插入工具调用标记、思考过程、错误提示、分段标识怎么办？**  
我的做法不是把所有东西都当纯文本吐给前端，而是设计统一事件协议，比如：

- `event: token`
- `event: tool_start`
- `event: tool_end`
- `event: thought`
- `event: warning`
- `event: done`

每个事件都带结构化字段，比如：

```json
{
  "request_id": "xxx",
  "seq": 12,
  "type": "tool_start",
  "tool_name": "getUserLastOrders",
  "payload": {}
}
```

前端按 `type` 分别渲染文本、工具状态条、错误提示或折叠式思考过程，这样不会把所有东西混成一段脏文本。

第四，**多轮对话 + 流式输出，怎么保证不乱序、不丢上下文？**  
关键是 `conversation_id + turn_id + seq` 三层标识。  
一个会话一条 `conversation_id`，每轮问题一个 `turn_id`，每个流片段一个 `seq`。这样无论跨服务透传还是前端重组，都能保证“同轮内按 seq 排序，不同轮按 turn_id 区分”。  
上下文不要靠前端拼，要在服务端基于 `conversation_id` 管理记忆和状态。

如果是 `Java/Go 网关 + Python 模型服务` 的跨服务流式透传，我会这样做：

1. Java/Go 接业务请求、鉴权、生成 `request_id`
2. 下游以 SSE 或 chunked HTTP 调 Python 模型服务
3. 中间层逐段读取上游 chunk，补充统一埋点和事件结构，再透传给前端
4. 一旦前端取消或中断，网关同时关闭与 Python 的上游流

所以这类题最好的答法不是某一个细节点，而是：**流式输出要有任务状态、事件协议、序号、断线恢复、取消机制和跨服务透传模型。**

### 相关知识点补充

- `request_id` 解决追踪
- `turn_id` 解决多轮隔离
- `seq` 解决乱序、重复和断点续传
- 流式协议最好是事件化，不要只传纯文本

### 高频追问

`Q：断点续打一定要真的继续原模型推理吗？`

**回答：**

不一定。更常见的是“恢复未消费的历史输出”，而不是让模型从中间状态继续推理。真正从中间继续推理很难，通常只在自控推理引擎和特殊状态保存下才可做。

`Q：如何避免资源泄露？`

**回答：**

要在连接关闭、用户取消、模型异常、超时四个路径下都统一做 finally 清理，包括上游连接、缓冲区、任务状态和上下文对象。

## 9. 高并发流式输出（QPS 1000+）下，SSE 长连接怎么治理？怎么防 OOM、做心跳、超时、安全截断、Token 计费和全链路埋点？

### 5 分钟答法

这个题其实不是在问“你会不会 SSE”，而是在问你有没有把流式输出当成一个**长连接系统**来治理。

高并发下，最大的风险有四类：

1. 连接太多，占满线程、内存和 FD
2. 慢客户端导致输出缓冲堆积
3. 上游模型慢或卡住，连接长期悬挂
4. 流式过程中缺乏安全和计费控制

我一般会从五层来做。

### 第一层：连接模型

不要用“一个连接一个阻塞线程”的老思路，而是尽量用异步 I/O 或响应式链路，让连接占用尽量轻。  
同时给每个连接加：

- 建连时间
- 最近活跃时间
- 心跳时间
- 当前任务状态

心跳可以是服务端每隔一段时间发一个轻量事件，比如 `event: ping`，前端收到即可忽略渲染，但能帮助代理层和客户端维持连接活性。

### 第二层：超时和释放

必须有几个超时：

- 建连超时
- 首 token 超时
- chunk 间隔超时
- 总任务超时

一旦超时，服务端要主动关闭连接，不能让无效连接无限挂着。  
这也是防 OOM 的关键点之一，因为很多内存泄露本质是“任务早就没意义了，但连接和缓冲还在”。

### 第三层：缓冲控制

SSE 下最怕慢客户端。服务端如果源源不断写，而客户端消费很慢，缓冲就会堆积。  
所以我会做：

- 每连接输出缓冲上限
- 单任务最大输出长度
- 慢消费者检测
- 超阈值直接降级或断流

不要默认认为“客户端一定跟得上”。

### 第四层：安全与实时截断

流式场景里，内容安全不能只在最后一次性审核。因为敏感词可能在中途就已经输出给用户了。  
所以我的做法是对 chunk 做实时检查：

1. 每吐一段文本先过轻量规则引擎或敏感词检测
2. 检测到高风险内容立即停流
3. 将 `stream_task` 标记为 `blocked`
4. 清理当前上下文缓存，避免后续恢复时继续输出违规内容
5. 前端收到一个结构化 `event: blocked`

这样才能做到真正的“实时拦截”。

### 第五层：Token 统计与计费

流式计费不能只在前端靠字符数估算，必须服务端统一统计。  
通常做法是：

- 每个 chunk 到达时累计 `output_token_estimate`
- 流结束时以上游模型返回的最终 usage 为准做校准
- 中间如果工具调用、重试，也分别记账

最终按：

- `request_id`
- `biz_line`
- `scene_code`
- `model_version`

做聚合，这样才能接成本报表。

### 全链路埋点怎么做

我会至少记录这些事件：

- 请求进入时间
- 上游模型首 token 时间
- 每个 chunk 输出时间
- 工具调用开始结束
- 用户取消
- 内容拦截
- 结束状态

然后统一打到链路追踪或日志系统里。  
这样后续才能分析：

- 首 token 延迟
- 平均 chunk 间隔
- 中断率
- 安全拦截率
- 单请求成本

如果结合你的当前项目来讲，虽然现在还没做到 1000+ QPS 的大规模 SSE 服务，但你完全可以把这套思路作为“如果把现在的 AI 点餐 Agent 做成生产级流式对话服务，我会这样扩展”的答法。

### 相关知识点补充

- 长连接服务的核心不是协议，而是连接生命周期治理
- OOM 常见原因：慢消费者、无超时、缓冲无限增长、异常路径未清理
- 首 Token 延迟、总时延、chunk 间隔、取消率都是关键指标

### 高频追问

`Q：SSE 高并发一定要上响应式框架吗？`

**回答：**

不绝对，但如果连接数非常高，阻塞式线程模型成本会越来越大。是否上响应式，要看连接数、语言运行时和团队栈。

`Q：流式输出怎么做内容审核又不太影响延迟？`

**回答：**

通常是“轻量规则前置 + 重模型抽检或高风险补充”，不能每个 chunk 都跑一个重模型，不然延迟会非常差。

## 10. Agent 的 Plan → Act → Observe → Reflect 在生产里怎么落地？ReAct 怎么避免“思考和行动脱节”？

### 5 分钟答法

我会先说一个很关键的判断：**在生产里，Agent 不能只停留在 prompt 里写一段“请你先思考再行动”，而必须把执行环路显式化。**

Plan → Act → Observe → Reflect，本质上是一个状态机：

1. `Plan`：理解用户目标，拆子任务
2. `Act`：选择工具并执行动作
3. `Observe`：读取工具结果或环境反馈
4. `Reflect`：判断结果是否足够、是否失败、是否需要调整下一步

如果把它落成生产级系统，我会把每一步都变成明确状态，而不是让模型在上下文里自由漂：

- `task_id`
- `step_id`
- `current_phase`
- `input_snapshot`
- `tool_name`
- `tool_args`
- `tool_result`
- `phase_status`
- `next_action`

这样做的好处是：

- 过程可观测
- 可重放
- 可中断
- 可人工接管

ReAct 框架的核心问题是“Reason”和“Act”很容易脱节。  
模型可能想得很好，但行动选错工具；也可能思考里说要查订单，实际却去查物流。  
为了避免这种脱节，我会做四件事：

### 1. 工具候选收窄

不要把所有工具都一次性暴露给模型，而是先做意图识别或路由，只给它最相关的一小组工具。  
这样可以显著减少“想的是 A，调的是 B”。

### 2. 强结构化中间结果

Reason 阶段不要只让模型输出自由文本，而是输出结构化计划，比如：

```json
{
  "intent": "查询用户最近订单偏好",
  "required_info": ["user_id"],
  "next_tool": "getUserLastOrders"
}
```

这样就能在执行前做校验，而不是让模型一句自然语言直接驱动行动。

### 3. 行动前校验

在真正执行工具前，工程侧先检查：

- 工具是否在候选列表里
- 参数是否齐全
- 参数类型是否匹配
- 当前状态下是否允许这个动作

这一步其实就是把“模型推理”和“系统执行”之间加一层守门。

### 4. Reflect 不只是让模型写一句“我再想想”

真正的 Reflection 应该基于执行结果来判断：

- 工具是否超时
- 返回是否为空
- 是否命中约束
- 是否出现重复调用

比如 `Observe` 发现订单工具返回空列表，那 `Reflect` 不能继续凭空推荐，而应该改成“退回菜单语义推荐”或者“补问用户更多条件”。

如果结合你的当前项目，虽然现在还不是完整 LangGraph 风格状态图，但你已经有一个轻量版环路了：

- `Plan`：模型理解用户点餐意图
- `Act`：触发菜单检索或工具调用
- `Observe`：读取检索结果和工具返回
- `Reflect`：决定是否继续调用工具，还是直接输出推荐

所以面试里我会说：当前项目落的是轻量版单 Agent 环路；如果往复杂业务演进，我会把它显式状态机化，用更强的调度和回退机制把环路固化下来。

### 相关知识点补充

- ReAct 最大的问题不是“不会推理”，而是“推理结果和执行动作断裂”
- 生产里 Agent 环路最好状态显式化，而不是全放在 prompt 里
- Reflection 本质是“基于反馈调整决策”，不是多写一段思考文本

### 高频追问

`Q：Act 失败怎么办？`

**回答：**

先区分是工具错误、参数错误还是权限错误。工具错误可重试或切降级路径；参数错误通常回到 Plan/Reflect 重新生成；权限错误则直接终止并给出受控回复。

`Q：Observe 没结果怎么办？`

**回答：**

不能默认等于“没有数据”，要区分“查询为空”和“工具失败”。前者可以继续推理，后者通常要做 fallback 或重试。

## 11. Agent 工具调用的 Schema 怎么设计？如何保证选对工具、参数不缺失、多步依赖不死循环？

### 5 分钟答法

Agent 工具设计里，Schema 是核心，不是附属品。  
因为大模型不是直接调 Java 方法签名，它实际上是在“理解一个结构化能力接口”。  
如果 Schema 设计不好，模型就会：

- 选错工具
- 少传参数
- 参数类型错误
- 重复调用
- 陷入循环

我一般把工具 Schema 设计分成四层。

### 第一层：工具职责要单一

一个工具只干一件事。  
不要设计“查用户+查订单+算推荐”这种大杂烩工具，否则模型很难准确选择。  
像你项目里当前的工具划分就比较自然：

- 查用户历史订单
- 查当前在售菜品
- 查当前在售套餐

每个工具语义清晰，模型更容易选对。

### 第二层：参数要强约束

Schema 至少要包含：

- 参数名
- 参数类型
- 是否必填
- 枚举范围
- 示例
- 参数语义说明

比如：

```json
{
  "name": "getUserLastOrders",
  "description": "查询指定用户最近订单，用于分析口味偏好",
  "parameters": {
    "type": "object",
    "properties": {
      "userId": {
        "type": "integer",
        "description": "当前登录用户ID"
      }
    },
    "required": ["userId"]
  }
}
```

这样可以显著减少参数缺失和类型错误。

### 第三层：工具选择不要完全放给模型

我通常会做“候选工具集限制”。  
比如当前问题如果明显是菜单问答，就只开放菜单相关工具；如果是订单售后，再开放订单和物流工具。  
这样能减少误选。

### 第四层：多步依赖要显式管理

像“查用户 → 查订单 → 查物流”这种链条，不能每一步都靠模型重新猜。  
最好的方式是引入任务状态和依赖图，明确：

- 当前已完成步骤
- 当前可执行步骤
- 上一步输出是什么
- 哪些参数可以复用

同时要加两个保护：

1. 最大步数限制  
2. 工具调用去重

比如同一个 `request_id` 下，如果相同工具 + 相同参数已经调用过一次，就不要无限重试，除非显式进入 retry 分支。  
这就是避免死循环的核心。

如果结合你的项目来讲，现在还是轻量单 Agent，工具数不多，所以问题还没完全暴露出来。  
但一旦工具数扩到十几个以上，Schema、候选收敛、参数校验、调用去重就会成为真正的工程问题。  
所以面试里我会强调：**工具不是“注册几个函数”就结束了，关键是 Schema 设计、参数约束、候选路由和依赖管理。**

### 相关知识点补充

- 工具数量少时，单 Agent + 明确工具描述就够用
- 工具数量一多，必须做工具路由和去重治理
- Schema 既是给模型看的，也是给工程系统校验用的

### 高频追问

`Q：如果模型总是漏传参数怎么办？`

**回答：**

要从三层解决：Schema 强约束、调用前工程校验、缺参数时触发补问或参数补全逻辑，不能直接把错误请求打到工具层。

`Q：如何避免重复调用同一个工具？`

**回答：**

对 `request_id + tool_name + normalized_args` 做幂等键，记录最近一次调用结果，相同调用直接复用或限制重试次数。

## 12. Agent 的短期记忆、长期记忆、身份传递、幂等、人工接管、可回放和可观测性，生产里怎么统一设计？

### 5 分钟答法

这道题其实考的是：你有没有把 Agent 当成一个**长期在线的任务系统**来看，而不是一轮 prompt 交互。

我一般会从六个维度统一设计。

### 1. 短期记忆

短期记忆解决的是“当前会话最近几轮在说什么”。  
你的项目里现在用的是窗口记忆，这是很合理的第一版。  
生产里我会把短期记忆存成：

- `conversation_id`
- `turn_id`
- `role`
- `content`
- `summary`
- `created_at`

短期记忆一般按时间顺序快速读取，适合放在 Redis、会话存储或高性能 KV 中。

### 2. 长期记忆

长期记忆不是把所有对话原文无限保存，而是抽取稳定偏好和长期事实，比如：

- 用户口味偏好
- 常点套餐
- 忌口
- 预算区间

长期记忆更像结构化用户画像，适合落数据库或向量库，而不是全塞上下文。

### 3. 身份传递

Agent 调工具时不能丢失身份，也不能把敏感登录态直接暴露给模型。  
我的做法是：

- 模型只看到“当前用户可访问哪些能力”的抽象
- 真实 `userId / role / permission scope` 由后端在工具层注入
- 工具层做权限校验，而不是相信模型传参

这点非常重要，因为否则会出现“模型构造越权参数”的风险。

### 4. 幂等

多个用户或同一用户重复点击，很容易重复触发 Agent 任务。  
所以我会对：

- 工具调用
- 外部接口
- 写库操作

都设计幂等键。  
比如 `request_id`、`conversation_id + turn_id`、或者业务唯一键。

### 5. 可回放和人工接管

生产里 Agent 一定会出错，所以必须支持回放和人工接管。  
要做到这一点，就必须记录：

- 每一步思考摘要
- 工具调用参数和结果
- 状态迁移
- 中间错误
- 最终回复

客服场景下，一旦 Agent 连续失败或触发高风险规则，就可以把当前会话和上下文快照直接转给人工。

### 6. 可观测性

如果从生产设计来说，当然要把 Agent 每一步都打点：

- reasoning time
- tool time
- retrieval time
- token cost
- error type
- retry count

这样你才能知道问题到底出在模型、检索、工具还是调度上。

但如果严格对应我当前项目的实现，我现在已经做了一个**第一版可观测性**：

- 在 `AgentController` 前面增加了一层 `AgentChatService`
- 每次 AI 对话都会生成 `requestId`
- 日志里会记录 `userId`、`memoryId`
- 会记录输入长度、输出长度、耗时和异常路径
- 缓存侧还补了 `LongAdder` 命中/未命中指标

也就是说，我现在不是完全没有观测，而是已经把最基础的“请求级链路日志 + 缓存指标统计”做起来了。  
如果后面继续增强，我会再往下补：

- retrieval time
- tool time
- token usage
- tool error type
- retry count
- answer quality 标注

所以这道题的核心不是一个个点分开答，而是统一答成：  
**短期记忆管当前轮次，长期记忆管用户画像；身份在工具层传递；幂等保护重复执行；当前项目已经有请求级日志和缓存指标，后续再往全链路 Agent 可观测性扩展。**

### 相关知识点补充

- 长期记忆不等于完整历史聊天原文
- 身份要“后端注入 + 工具层校验”，不能直接相信模型
- 没有可回放能力的 Agent，线上排障会非常痛苦

### 高频追问

`Q：人工接管时最少要保存什么？`

**回答：**

至少保存当前会话摘要、用户原始问题、最近几轮上下文、已调用工具及结果、当前 Agent 状态和最后一个失败原因。

`Q：为什么不能把所有历史聊天都塞进上下文？`

**回答：**

因为成本太高、噪声太大，而且很多历史内容并不是稳定偏好。长期信息应该抽成结构化记忆，而不是无限堆 token。

---

## 13. LangChain 和 LangGraph 在生产里怎么选？开源 Agent 框架常见坑有哪些？

### 5 分钟答法

我会把这题答成“复杂度和控制力”的取舍。

如果业务还处于单 Agent、少量工具、轻量 RAG、流程相对线性的阶段，我更倾向用 LangChain 这一类组件化框架，开发效率高，上手快，适合先把链路跑通。  
你现在这个项目，其实就更接近这个阶段：单 Agent + 检索 + 工具调用 + 记忆，没有特别复杂的分支状态机，所以 LangChain4j 这种抽象是合适的。

什么时候考虑 LangGraph？  
当流程明显变成图结构时，比如：

- 审批流
- 工单处理
- 多步售后
- 复杂多工具协作
- 失败回退和人工接管

这时仅靠 prompt 和线性链路已经不够，你需要显式状态图、节点、边、分支、回退和恢复机制，LangGraph 这种图式框架就更适合。

开源 Agent 框架在生产里最常见的坑，我总结有六类：

1. 过度魔法化  
   本地 demo 很丝滑，生产里一出错很难定位到底哪一步坏了。

2. 状态不透明  
   没有显式状态机，执行过程全在黑盒里，排障困难。

3. 工具调用无限制  
   工具过多、无候选裁剪、无最大步数、无去重，容易形成循环调用。

4. 日志和埋点不完善  
   只能看到最终回复，看不到中间 reasoning、tool use、retrieval。

5. 框架升级和兼容问题  
   Agent 框架变化很快，生产里不能把所有控制权都押在一个版本不稳定的开源库上。

6. 过早上复杂框架  
   业务还很简单，却先引入多 Agent 图编排，导致系统复杂度远大于收益。

所以我一般的建议是：

- 简单场景先用 LangChain 类方案把链路跑通
- 一旦流程开始图化、分支化、需要恢复和回退，再考虑 LangGraph
- 无论用什么框架，状态、日志、幂等、超时、限流、人工接管这些生产能力一定要自己掌握，不能全依赖框架默认行为

这也是我对当前项目的理解：现在更适合轻量链路；如果未来扩展到售后、客服、营销、工单和审批协同，LangGraph 或显式状态机会更合适。

### 相关知识点补充

- 框架选型不是看“谁最火”，而是看业务流程形态
- 生产里最贵的是可控性，不是 demo 效果
- 状态、日志、超时、幂等必须掌握在自己系统里

### 高频追问

`Q：AutoGen、MetaGPT 这类框架为什么线上坑多？`

**回答：**

因为多 Agent 看起来强大，但任务边界、通信成本、调度和结果可控性都更复杂，很多时候业务还没复杂到需要它们，过早引入只会增加不稳定性。

`Q：LangGraph 最适合什么场景？`

**回答：**

最适合流程天然是状态机或图结构的业务，比如审批、工单、多步售后、多角色协作，而不是简单问答式 Agent。

---

## 14. 百万级文档 RAG、延迟 <200ms，索引、分片、缓存、增量更新、混合检索、重排和评估体系怎么设计？

### 5 分钟答法

如果把当前项目从菜单级 RAG 扩展到**百万级文档、检索延迟 <200ms** 的生产系统，我不会把它理解成“换个向量库就行”，而会把它看成一个**在线检索系统 + 异步索引系统 + 质量评估系统**的组合题。

我会先定三个目标：

1. 检索层延迟稳定在 `<200ms`
2. 文档新增、修改、删除能做增量更新，不能靠全量重建
3. 检索效果不能只看有没有结果，而要看召回率、重排效果和最终回答可信度

所以我会把系统拆成七层：

### 第一层：知识加工层

文档进入系统后，不是直接做 Embedding，而是先做：

- 文档解析
- 结构恢复
- 清洗去噪
- 分块
- 元数据抽取
- 去重和版本管理

这里最关键的是两件事：**分块**和**元数据**。  
分块决定召回粒度，元数据决定后面能不能做过滤、权限、时间范围约束。

对于百万级文档，我通常会优先用：

- 结构优先切分
- 必要时父子分块
- 每个 chunk 挂上 `doc_id / chunk_id / version / tenant / type / time_range / permission_scope`

因为真正上线后，检索不只是“语义相似就行”，还要满足权限、时效和业务范围约束。

### 第二层：索引层

百万级下我不会只靠单一种类索引，而是至少两套：

- 稀疏索引：倒排索引，承接 BM25
- 稠密索引：ANN 向量索引，承接语义召回

也就是典型的**混合检索**。

为什么一定要混合？  
因为稠密检索擅长语义相似，但对精确实体词、缩写、产品名、错误拼写、术语别名不一定稳；  
BM25 这种稀疏检索对关键词命中更强，但对语义改写能力弱。  
两者混合，才能在实际业务里把召回率和稳定性一起拉起来。

### 第三层：在线检索层

在线查询时，我会把流程拆成：

1. Query 预处理 / 改写
2. 稀疏召回
3. 稠密召回
4. 结果融合
5. 元数据过滤
6. Rerank
7. 上下文压缩
8. 返回给 LLM

这里面真正影响延迟的不是某一个点，而是**每一步都要有预算**。

比如我会粗分预算：

- Query 改写：10~20ms
- 稀疏检索：20~40ms
- 稠密检索：30~60ms
- 融合 + 过滤：10~20ms
- Rerank：30~50ms
- 序列化与拼装：10ms 左右

总的才有机会压到 200ms 内。

### 第四层：缓存层

缓存不能只做一层，我通常会做四层：

- Query 改写缓存
- Query Embedding 缓存
- 检索结果缓存
- Rerank 结果缓存

因为在生产里，同类问题会反复出现，尤其是企业知识库、客服、SOP、制度说明这种场景。

### 第五层：增量更新层

百万级文档场景最怕“知识变了，但索引还是旧的”。  
所以我会明确：

- 数据库 / 文档主存储是真相源
- 向量索引和倒排索引都是副本

增量更新我一般会做成：

- 文档主表
- 文档版本表
- 索引任务表
- CDC / 事件驱动同步
- 软删除或墓碑标记

也就是说，新增、更新、删除都必须变成可追踪事件，而不是直接改索引。

### 第六层：质量评估层

RAG 效果绝对不能只看最终回答像不像人话。  
我会拆成三层指标：

- 检索层
- 生成层
- 系统层

检索层看：

- Recall@K
- MRR
- NDCG
- Hit Rate

生成层看：

- Answer Relevancy
- Faithfulness
- Hallucination Rate
- 引用依据覆盖率

系统层看：

- P95 / P99
- QPS
- 缓存命中率
- 索引更新时延
- Rerank 耗时

### 第七层：灰度和回放层

生产级系统一定要能回放 query，比较不同索引参数、不同重排模型、不同 chunk 策略的效果。  
否则你根本不知道优化是不是在“拍脑袋”。

所以我会保留：

- 原始 query
- 改写后 query
- 召回候选
- 重排前后顺序
- 最终注入 LLM 的上下文
- 最终答案

这样离线评估、人工抽检、线上回归测试才能真正闭环。

所以这题如果压成一句话，我会说：

> 百万级 RAG 的核心不是“把文档向量化”，而是把它当成一个完整检索系统来设计：知识加工、混合索引、低延迟查询链路、多层缓存、增量更新、一致性控制、重排和评估体系要一起做。

---

### 底层原理拆解

下面这部分如果面试官往底层原理深挖，我会继续往下讲。

### 14.1 索引怎么设计？为什么单纯暴力向量扫描不行？

如果是百万级文档，每个 chunk 一条向量，假设是 768 维 `float32`：

- 单条向量大小大约是 `768 * 4 = 3072 bytes`
- 100 万条光向量原始数据就接近 3GB
- 这还不算索引图结构、元数据、倒排索引、缓存和冗余副本

如果每次查询都做暴力全量扫描，本质是：

1. 把 query 向量和所有候选做相似度计算
2. 再做 topK 选择

这个复杂度太高，延迟会随着数据量线性放大。  
所以必须上 **ANN，Approximate Nearest Neighbor，近似最近邻检索**。

常见向量索引有三类思路：

#### 1. HNSW

HNSW 本质上是**分层小世界图**。  
每个向量是图里的一个节点，节点之间连接“近邻”。  
查询时从高层稀疏图开始快速逼近，再往低层密集图逐步细化，最后找到近似最近邻。

它的特点是：

- 召回率高
- 查询快
- 适合读多写少
- 内存开销大
- 动态插入比 IVF 更灵活

所以如果数据规模是百万级、读多写少、内存够，我会优先考虑 HNSW。

#### 2. IVF

IVF 是**先聚类，再局部搜索**。  
它会先把向量空间分成很多簇，查询时先找到离 query 最近的几个簇，再只在这些簇里搜索。

它的好处是：

- 查询更快
- 更适合更大规模数据

但缺点是：

- 参数调不好会漏召回
- 动态更新和聚类质量相关

#### 3. PQ / OPQ

PQ 是**乘积量化**。  
本质上是把高维向量切成多个子空间，每个子空间分别量化成较短编码。  
这样可以显著降低存储和计算成本。

好处是：

- 存储成本低
- 检索更快

代价是：

- 精度有损
- 召回率会下降

所以真正生产里常见是：

- `IVF_FLAT`
- `IVF_PQ`
- `HNSW`

这些组合，本质是在**速度、内存、召回率**之间做权衡。

---

### 14.2 BM25 到底是怎么检索的？为什么它对混合检索仍然重要？

BM25 本质上是基于词法匹配的打分函数。  
它关心三件事：

1. 词项在当前文档出现得多不多，也就是 TF
2. 这个词在全库里稀不稀有，也就是 IDF
3. 文档是不是太长，需要做长度归一化

它不是简单“词频越高分越高”，而是会抑制词频无限增长，也会惩罚超长文档。  
所以它特别适合：

- 精确实体词
- 专业术语
- 产品名
- 缩写
- 菜名、套餐名、型号名

比如用户问：

- “HNSW 和 IVF 区别”
- “宫保鸡丁”
- “请假审批 SOP”

这类 query 如果只靠向量检索，不一定稳；BM25 往往能非常快地把包含关键术语的文档召回来。

所以混合检索不是“两个都上看起来高级”，而是因为：

- 稠密检索解决语义相似
- 稀疏检索解决术语、关键词和精确匹配

两者缺一不可。

---

### 14.3 混合检索怎么融合？为什么不能简单把两边结果拼起来？

如果稀疏检索和稠密检索都返回 topK，直接拼起来的问题是：

- 两边分数分布不在一个量纲
- 有的 query 明显更依赖关键词
- 有的 query 明显更依赖语义

所以融合一般有三种常见方式：

#### 1. 归一化加权

把两边分数做归一化，再按权重相加。  
适合分数分布相对稳定、业务比较可控的场景。

#### 2. RRF

Reciprocal Rank Fusion，本质是按排名融合，不强依赖原始分数。  
它常见且鲁棒，因为不同检索器分数难以直接比较，但排名通常更稳定。

#### 3. LTR / 学习排序

把稀疏分、稠密分、时间、点击率、文档类型、权限等特征一起喂给排序模型。  
这是更生产级的做法，但工程复杂度也更高。

如果是我来做一版可落地系统，我通常会：

- 第一版先上 RRF
- 稳定后再考虑加权融合
- 再往后如果在线反馈足够，再上学习排序

---

### 14.4 分片怎么做？为什么分片不是只看数据量？

分片至少有三种维度：

#### 1. 按租户 / 业务域分片

比如企业知识库、客服知识库、运营文档、产品文档本来就是不同语料域。  
按业务域分片好处是：

- 缩小候选集合
- 减少跨域噪声
- 权限边界更清晰

#### 2. 按时间 / 生命周期分层

热文档和冷文档分层：

- 热索引放内存、低延迟
- 冷索引放更便宜存储，按需召回

#### 3. 按索引规模水平切分

也就是常规的数据水平分片。  
底层可以是：

- Elasticsearch shard
- Milvus collection / partition / segment
- 自己做 hash / range 路由

分片不能只看“数据量大不大”，还要看：

- 查询模式是否局部化
- 权限是否天然隔离
- 热点是否集中
- 更新是否集中在某些分片

因为很多时候真正的问题不是“总量大”，而是“热点特别偏”。

---

### 14.5 缓存到底要缓存什么？为什么不能只缓存最终答案？

RAG 系统里缓存最常见的误区就是只想缓存最终答案。  
但实际上，最终答案最容易失效，而且最容易受上下文和模型版本影响。

我更建议缓存下面几层：

#### 1. Query 改写缓存

同一类问题改写结果相对稳定，缓存价值高，成本低。

#### 2. Query Embedding 缓存

同一个 query 重复出现时，不需要每次都重新做 embedding。

#### 3. 检索结果缓存

特别适合 FAQ 型和热点问法。

#### 4. Rerank 结果缓存

如果 query 重复率高，重排也是可缓存的。

#### 5. 最终答案缓存

这个可以做，但要更谨慎，通常需要带上：

- query 标准化结果
- 知识库版本
- 权限范围
- 模型版本

否则特别容易答非所问或者返回旧知识。

所以我的原则是：

> 越靠前的中间结果越适合缓存，越靠后的最终结果越要慎重缓存。

---

### 14.6 增量更新怎么做？为什么数据库和索引的一致性这么难？

因为索引天然是副本，不是真相源。  
只要数据库和索引不是一个事务里同时提交，就一定存在不一致窗口。

我一般会这样设计：

#### 1. 真相源

数据库 / 对象存储里的原始文档是真相源。

#### 2. 事件驱动

文档新增、修改、删除先写主存储，再产生事件。  
事件可以来自：

- 业务代码直接发消息
- Outbox 表
- Binlog CDC

#### 3. 索引任务表

索引系统不要直接“收到消息就改”，而是先写一张索引任务表，记录：

- 文档 ID
- 版本号
- 变更类型
- 当前状态
- 重试次数

#### 4. 版本控制

每个 chunk、每个文档都带版本号。  
检索时只认当前生效版本。

#### 5. 软删除 / 墓碑

删除不是立刻物理删，而是先打 tombstone。  
这样即使索引清理有延迟，也不会被继续召回。

#### 6. 后台 compaction

等索引和存储都稳定后，再异步清理旧版本。

所以这题的底层本质是：

> 数据库和索引不是强一致事务，而是“主存储 + 索引副本”的最终一致系统；要靠版本号、状态位、任务表和重试机制把一致性补出来。

---

### 14.7 Rerank 为什么有效？为什么它比直接多召回几个更靠谱？

因为召回阶段的目标是“别漏”，重排阶段的目标是“别乱”。  
两者优化目标不同。

向量检索和 BM25 召回出来的是候选集合，它更关注 Recall。  
但真正交给 LLM 的上下文不能太多，否则：

- token 成本高
- 噪声大
- 容易幻觉

所以需要 Rerank 把候选集合再压成一个更干净的 topN。

常见 Rerank 模型原理是：

- 把 query 和候选文档拼成一对输入
- 用 cross-encoder 重新打分

它比 bi-encoder 更准，因为 query 和候选会在同一个 Transformer 里做交叉注意力。  
但代价就是更慢。

所以工程上我通常会这样做：

- 召回 top50
- rerank 到 top5 或 top10
- 最后再做上下文压缩后给 LLM

这样比“直接召回 top100 全塞进去”效果通常更稳。

---

### 14.8 评估体系怎么做才算真正闭环？

我会把评估分成离线、在线和人工三层。

#### 1. 离线评估

构造 query-黄金文档对，评估：

- Recall@K
- MRR
- NDCG
- Precision@K

如果这层不过，生成层再好也没意义。

#### 2. 在线评估

看系统实时指标：

- P95 / P99
- cache hit rate
- query rewrite 命中率
- rerank 耗时
- 索引更新时延
- 降级比例

#### 3. 人工评估

人工抽样检查：

- 答案是否相关
- 是否忠于证据
- 是否引用错误知识
- 是否遗漏关键约束

如果再往上做，我会把用户点击、采纳、追问、纠错这些反馈反哺成在线评估数据。

所以我会强调：

> 没有评估集和回放机制的 RAG 优化，很容易变成“今天调参数觉得好了，明天换一批 query 又坏了”。

---

### Embedding、相似度和 Fine-tuning 取舍

Embedding 模型选型上，我会优先看三件事：

1. 语种适配
2. 向量维度和速度
3. 域内效果

不是维度越高越好，因为维度高意味着：

- 存储更贵
- 索引更重
- 计算更慢

相似度上：

- 余弦相似度更关注方向，适合语义相似
- 点积常用于归一化向量或特定索引实现
- 欧氏距离在高维语义检索里相对少一些

RAG 和微调怎么选？  
我会优先 RAG，因为知识变化快、事实动态，RAG 的更新成本更低；  
只有当问题不是“缺知识”，而是“缺行为模式、输出格式、领域决策风格”时，我才会考虑微调。

所以这道题最完整的答法应该是：  
**知识加工、ANN 索引、BM25 稀疏检索、混合融合、分片路由、多层缓存、事件驱动增量更新、Rerank 和评估闭环一起讲，才像真正的生产级 RAG 设计。**

### 相关知识点补充

- 百万级 RAG 的核心瓶颈通常不在 LLM，而在检索系统
- HNSW 更像高召回高内存方案，IVF/PQ 更像高压缩高吞吐方案
- BM25 的价值在术语、实体词和精确匹配
- 一致性难点来自“主存储”和“索引副本”分离
- 缓存最值得做的是 query、embedding、检索结果和 rerank 结果，不只是最终答案

### 高频追问

`Q：为什么不能直接用一个大模型把所有文档都塞进去？`

**回答：**

因为上下文窗口再大也不是无限的，而且越长越贵、噪声越大。RAG 的本质就是先用检索把候选空间缩小，再把最相关的证据交给模型，而不是让模型在海量原始文档里自己找。

`Q：为什么 HNSW 查得快？`

**回答：**

因为它不是线性扫所有向量，而是在分层近邻图里做贪心逼近搜索。高层图负责快速逼近目标区域，低层图负责精细化找近邻，所以查询复杂度远低于暴力全量扫描。

`Q：为什么同一问题还要做缓存，不怕旧知识吗？`

**回答：**

要结合知识版本和 TTL。缓存 key 里最好带知识库版本号、租户、权限范围和模型版本，文档更新后旧缓存自然失效。缓存不是永久保存，而是为了减少重复 query 在短时间窗口内的重复计算。

`Q：检索结果很多但回答跑偏，最先优化哪一层？`

**回答：**

先看召回集合是不是太脏。通常会先查元数据过滤和 rerank，再看 chunking 和 query rewrite；如果候选本身就不对，继续调 Prompt 基本治标不治本。

`Q：百万级下，最容易踩的坑是什么？`

**回答：**

最容易踩的坑不是“向量库选错了”，而是没有把版本、一致性、缓存、评估和回放体系补齐。很多系统离线看不错，一上生产就被旧知识、噪声召回、权限串数据和评估缺失拖垮。

## 15. LLM 工程化里，QPS 100+ 或更高时，排队、削峰、批量推理、多模型调度、量化、熔断、GPU 资源隔离怎么设计？

### 5 分钟答法

这题本质上是在问：**你是否把 LLM 服务当成一个真正的高并发在线推理系统来治理。**

我会按五层来答。

### 第一层：入口治理

峰值 QPS 100+ 甚至更高时，不能让请求直接无序打到模型。  
入口层要做：

- 限流
- 排队
- 业务优先级
- 用户优先级

比如免费用户和付费用户的请求不能同优先级；超长上下文和普通请求也不能混着抢资源。

### 第二层：批量推理和调度

像 vLLM、TGI 这类推理服务，本质上都在做一件事：  
**把原本一条一条进来的请求，在 GPU 侧做更高效的调度和批处理。**

vLLM 的核心优势之一是连续批处理和更高效的 KV Cache 管理；  
TGI 偏 Hugging Face 生态，易用性强；  
TensorRT-LLM 更偏极致性能优化，但工程复杂度更高。  
怎么选，要看：

- 你是自研模型还是通用开源模型
- 更看重吞吐还是易维护
- 是否能接受更重的工程链路

### 第三层：多模型路由

生产里不是所有请求都配得上大模型。  
我会做模型路由：

- 简单问答、小任务走小模型
- 复杂推理、多工具协作走大模型
- 特定领域问题走专门模型

这样可以显著降低成本。

### 第四层：量化和资源隔离

量化的本质是用更低精度表示参数，换取更快推理和更低显存占用。  
INT8、INT4、FP8 都是在这个思路下的不同精度方案。  
但量化不是越狠越好，关键是平衡：

- 可接受精度损失
- 推理吞吐提升
- 显存占用下降

实际生产里通常会先做离线评估，看任务质量掉多少，再决定上多激进的量化方案。

GPU 资源隔离这块，我会做：

- 不同业务队列隔离
- 超时任务抢占或取消
- 大上下文请求和短请求分池
- 长任务异步化

### 第五层：熔断和备用模型

如果主模型 API 报错、限流或宕机，不能直接把用户体验打崩。  
我会做：

1. 上游超时和错误码识别
2. 熔断器防止继续打爆故障模型
3. 备用模型或降级模型切换
4. 给用户结构化提示，而不是直接 500

如果是你的项目延展版本，我会把轻量问答先切到便宜的小模型，复杂推荐才走强模型，这样既能控成本，也能提升整体可用性。

所以这类题的关键不是背 vLLM/TGI 名字，而是要讲清楚：

- 请求如何排队
- 模型如何调度
- GPU 如何隔离
- 故障如何切换
- 成本如何控制

### 相关知识点补充

- 高并发 LLM 系统不只是模型快不快，更重要的是调度和治理
- 多模型路由是成本控制的核心手段之一
- 异步长任务要和同步短任务隔离

### 高频追问

`Q：如果用户恶意构造超长上下文怎么办？`

**回答：**

入口就要限长度、限 token、限频率；并在鉴权层和业务层做额度控制，不能等请求进到模型侧再处理。

`Q：量化后最常见的坑是什么？`

**回答：**

某些任务质量掉得比预期大，尤其是复杂推理、工具调用格式和长上下文稳定性，所以一定要在业务指标上做充分离线评估，而不是只看 benchmark。

## 16. Prompt Injection、脱敏、输出审核、成本控制、私有部署 vs API、可解释可审计，这些安全与成本题怎么答？

### 5 分钟答法

我一般把这类题统一看成“AI 系统的安全治理和成本治理”。

### 先说 Prompt Injection

它的原理本质上是：用户输入混入了恶意指令，试图覆盖系统提示词、绕过工具权限或诱导模型泄露信息。  
生产级防御不能只靠一句“忽略用户中的恶意指令”，而要做多层防御：

1. 防御性 system prompt  
2. 输入侧规则拦截  
3. 工具层权限隔离  
4. 输出审核  
5. 高风险场景人工兜底

最关键的是：**不要让模型直接决定权限和敏感数据访问。**

### 再说敏感信息脱敏

像手机号、订单号、身份证号、地址这些敏感信息，不应该原样进模型。  
我的做法是：

- 进入模型前做脱敏或映射
- 工具层只接收必要字段
- 日志也做脱敏

比如订单号可以映射成临时 token，让模型只知道“订单A”，而不是看到真实号码。

### 输出审核

输出审核要分两层：

1. 轻量规则和敏感词前置
2. 高风险场景补充分类模型或人工审核

流式场景下还要做实时截断，不能等全文生成完再审。

### 成本控制

成本控制不是只看模型价格，而是组合治理：

- 限流
- 缓存
- 多模型路由
- RAG 缓解无效长上下文
- Tool 调用控制
- 批量推理

### 私有部署 vs API 调用

我会从四个维度比较：

1. 成本：调用量小或波动大时 API 更省；稳定高量时私有化可能更划算  
2. 稳定性：API 依赖外部厂商，私有部署可控但运维更重  
3. 安全性：敏感数据、高合规场景更偏私有化  
4. 迭代速度：API 调用更快，私有部署更重

### 可解释、可审计、可回溯

这件事不能靠“模型自己解释”，而要靠系统日志设计。  
我会记录：

- 输入摘要
- 检索结果
- 工具调用
- 中间状态
- 输出结果
- 风险标签
- 操作人或用户 ID

这样事后才能回放和审计。

所以这类题最好的答法是：**AI 系统的安全治理要做到输入防护、权限隔离、输出审核、日志可审计；成本治理要做到路由、缓存、限流、计费和模型策略。**

### 相关知识点补充

- Prompt Injection 的本质不是“模型中毒”，而是输入指令污染控制边界
- 敏感信息保护要覆盖：输入、工具、输出、日志四层
- 审计依赖结构化日志，不依赖模型自述

### 高频追问

`Q：为什么只靠 system prompt 不能防注入？`

**回答：**

因为 prompt 本质还是文本约束，不是硬权限控制。真正的权限边界必须在工具层、数据层和网关层实现。

`Q：成本暴增最常见原因是什么？`

**回答：**

超长上下文、无缓存、所有请求都打大模型、无重试控制、无恶意限流，这些组合起来最容易导致成本失控。

## 17. 如果让你设计一个企业知识库 Agent、Text2SQL Agent、客服多 Agent 或可上线的 Agent 平台，你会怎么回答？

### 5 分钟答法

这类设计题表面不同，其实底层框架高度复用。  
我会统一按“入口层、编排层、知识层、工具层、安全层、观测层”来答。

### 1. 入口层

提供统一 API / SDK / Web 页面，负责：

- 鉴权
- 限流
- request_id
- 场景识别

### 2. 编排层

负责：

- 单 Agent 或多 Agent 路由
- 状态机执行
- 超时控制
- 重试与回退
- 人工接管

### 3. 知识层

负责：

- 文档解析
- 分块
- Embedding
- 检索
- 重排
- 权限过滤

企业知识库 Agent 和客服 Agent 都强依赖这一层。

### 4. 工具层

负责和现有后端系统对接，比如：

- 查用户
- 查订单
- 查物流
- 建工单
- 调 CRM
- 发消息

Text2SQL Agent 也属于工具层的一种特殊形式，只不过工具是“生成并执行 SQL”。

### 5. 安全层

这是最容易被忽略但最关键的一层：

- Prompt 注入防护
- 数据权限
- 工具权限
- 输出审核
- SQL 沙箱
- 审计日志

比如 Text2SQL Agent 一定不能让模型直接执行任意 SQL，而是要：

- 限定库表白名单
- 做只读沙箱
- 禁止 DDL/DML
- 参数化执行
- 结果行数限制

### 6. 观测层

统一记录：

- 检索召回
- 工具调用
- Token 消耗
- 延迟
- 成功率
- 错误率
- 人工接管率

如果是企业知识库 Agent，我会重点强调：

- 文档权限过滤
- 多租户隔离
- 检索延迟
- 知识更新同步

如果是客服多 Agent，我会重点强调：

- 意图识别 Agent
- 知识库问答 Agent
- 工单生成 Agent
- 人工转接 Agent

如果是 Agent 平台，我会重点强调：

- Agent 注册
- 工具注册
- 模板和工作流管理
- 日志与评估面板
- 版本管理

所以这类设计题，最稳的答法不是被业务名词带着跑，而是把它们还原成同一个生产级 Agent 系统架构，然后根据具体场景突出重点。

### 相关知识点补充

- 企业知识库 Agent 的核心是权限过滤 + 检索质量
- Text2SQL Agent 的核心是 schema 理解 + SQL 安全
- 客服多 Agent 的核心是路由、接力、人工兜底
- Agent 平台的核心是“能力复用”和“工程治理”

### 高频追问

`Q：Text2SQL 最大风险是什么？`

**回答：**

最大风险不是 SQL 写错，而是越权查询和注入执行，所以一定要把执行环境做成只读、白名单、沙箱化。

`Q：客服多 Agent 为什么不能只用一个万能 Agent？`

**回答：**

因为意图识别、知识问答、工单流转、人工接管属于不同职责，拆开后更可控、更好评估，也更方便失败时局部兜底。

## 18. RAG 里文档切分有哪些策略？什么是自适应切分？你项目里用的是什么策略？

### 5 分钟答法

如果面试官问到 RAG 里的文档切分，我一般不会只答“按固定长度切”。  
因为切分其实是 RAG 质量里非常核心的一环，它直接影响三个结果：

1. 检索召回是否准确  
2. 上下文是否完整  
3. 最终送给 LLM 的内容是否既相关又不过长

所以我通常会先把切分策略分成几类来讲。

### 第一类：固定长度切分

这是最基础、最常见的一类策略。  
比如按 `N` 个字符、`N` 个词、或者 `N` 个 token 去切，每段之间再加一点 overlap。

它的优点是：

- 简单
- 实现成本低
- 对任意文本都能快速落地

缺点也很明显：

- 容易把一个完整语义从中间截断
- 对结构化文档不友好
- 有时候一段里塞了多个主题，召回会变脏

所以固定长度切分适合作为 baseline，但通常不是最优策略。

### 第二类：按结构切分

也就是优先按照文档自身的结构边界来切，比如：

- 标题
- 段落
- 小节
- 列表
- 表格
- 代码块

这种方式的优点是语义自然，比较符合人类阅读方式，也更容易保持上下文完整。  
比如企业知识库文档、技术文档、FAQ、Markdown 文档，按标题和段落切通常会比纯 token 切分效果更好。

但它也有边界：

- 某些段落可能特别长，直接按段落切会超出模型和检索的最佳粒度
- 某些结构边界虽然明显，但信息密度不均匀，可能导致 chunk 太大或太小

所以很多时候结构切分要和 token 上限结合。

### 第三类：语义切分

语义切分更强调“在语义变化的位置断开”，而不是在机械长度或段落边界断开。  
做法一般是：

- 先按句子或小段切成基础单元
- 再基于 embedding 相似度、主题变化、断句特征，把连续语义相近的单元合并成 chunk

它的优势是：

- chunk 内部语义更集中
- 检索相关性通常更好
- 适合长文本、混合主题文本

但代价是：

- 预处理更复杂
- 计算成本更高
- 工程实现比简单切分重

### 第四类：父子分块 / 分层切分

这是我非常认同的一类生产策略，尤其适合长文档。  
它的思路是：

- 小块负责检索，保证召回精度
- 大块或父块负责回填，保证上下文完整

比如先把文档按较小粒度切成子块用于向量检索，一旦某个子块被召回，再把它所属的父段、父章节、甚至整段上下文一起回填给 LLM。

这样做能解决一个很典型的问题：

> 小块检索准确，但给模型的信息不完整；大块信息完整，但检索时噪声太大。

父子分块就是在召回精度和上下文完整性之间做平衡。

---

### 什么是自适应切分

自适应切分，本质上不是某一种固定算法，而是一种策略思想：

> 不同文档、不同结构、不同长度、不同信息密度，用不同粒度切分，而不是全量文档都套一个固定 chunk size。

我一般会把自适应切分理解成三层自适应：

#### 1. 按文档类型自适应

不同文档天然适合不同策略：

- FAQ：适合按问答对切
- Markdown / 技术文档：适合按标题和段落切
- PDF 报告：适合先还原结构，再做段落切
- 表格：适合按行列、表头和单元语义切
- 代码：适合按函数、类、文件结构切

也就是说，文档类型决定初始切分方式。

#### 2. 按长度和 token 上限自适应

比如一个段落本身就很短，那就不需要再拆；  
如果一个段落非常长，就不能机械保留原结构，而要进一步按句子、语义或 token 上限再切一层。

所以自适应切分往往会有这样一个逻辑：

- 优先按结构切
- 如果超出阈值，再细分
- 如果太短，再向上合并

#### 3. 按语义密度自适应

有些内容虽然长度差不多，但信息密度完全不同。  
比如一个长表格说明段和一个叙述性自然段，适合的 chunk 粒度就不一样。

自适应切分会根据：

- 句子相似度变化
- 标题边界
- 关键词密度
- token 长度

综合判断应该在哪断开。

---

### overlap 应该怎么理解

切分时 overlap 也是一个经常被追问的点。

overlap 的目的不是“多复制一点内容”，而是为了防止信息恰好落在 chunk 边界处被截断。  
比如一个关键结论出现在上一段末尾和下一段开头的衔接位置，如果完全没有 overlap，就容易导致两边 chunk 都缺半句关键信息。

但 overlap 也不是越大越好：

- 太小，边界信息丢失
- 太大，冗余严重，检索噪声增加，存储和算力成本也上升

所以 overlap 要结合文档类型和 chunk 长度来选，通常是一个“保边界、不制造过多冗余”的平衡值。

---

### 我当前项目里的切分策略是什么

如果严格对应我现在这个外卖 Agent 项目，其实它和传统“长文档 RAG”不完全一样。

因为我当前检索的知识主体不是几万字的长报告，而是：

- 菜品
- 套餐
- 分类信息
- 价格
- 描述

也就是说，我这里的知识单元天然就比较短，而且一条菜品或套餐本身就是一个相对完整的语义对象。

所以我当前项目里，采用的不是复杂长文档切分，而更接近**“结构化对象级切分”**：

- 一道菜就是一个 chunk
- 一个套餐就是一个 chunk

然后把这个 chunk 组织成类似：

`菜名 + 分类 + 价格 + 描述`

这样的文本，再去做 embedding。

为什么这样设计？

因为在这个场景里，一条菜品信息本身就已经是最自然的最小检索单元了。  
如果再把一道菜的描述切成很多更小块，反而会把“名称、价格、描述、类别”这些强关联字段拆散，影响召回质量。

所以对我当前项目来说，最合理的策略其实不是固定 token 切分，而是：

> **按业务对象切分**

这是一种非常典型的“结构优先策略”。

---

### 如果把我这个项目扩展到企业知识库 / 长文档场景，我会怎么做

如果未来这个项目从菜单检索扩展到：

- 商家运营文档
- 客服 SOP
- 企业内部知识库
- 工单规则说明

那我就不会继续只按对象级切分了，而会采用一套更通用的自适应策略：

#### 第一步：按文档类型分流

- FAQ 按问答对切
- Markdown 按标题层级切
- PDF 先 OCR / 结构还原，再按段落切
- 表格按“表头 + 行”结构切

#### 第二步：按结构切分

先尊重标题、段落、列表、表格这些天然边界。

#### 第三步：做长度约束

如果某一段太长，再按句子或 token 上限做二次切分；  
如果某一段太短，则和相邻语义段合并。

#### 第四步：适当 overlap

保留一部分上下文，防止边界信息断裂。

#### 第五步：必要时做父子分块

检索用子块，生成时回填父块，提高答案完整性。

所以如果面试官问：

> “你的策略是什么？”

我会非常明确地答：

> 我当前项目里因为知识载体是菜品和套餐这样的短结构化对象，所以采用的是按业务对象切分，每个菜品或套餐就是一个独立 chunk；如果扩展到长文档场景，我会采用“结构优先 + 长度约束 + overlap + 必要时父子分块”的自适应切分策略，而不是全量固定长度切分。

---

### 为什么我不推荐一开始就用固定长度切分

因为固定长度切分最大的问题是“工程上简单，但语义上经常不优”。

你可以把它当作 baseline，但如果面试官追问生产级方案，我一定会强调：

- 文档结构要尽量保留
- 语义边界要尽量保留
- chunk 不能太大也不能太小
- 检索粒度和生成粒度最好分开看

真正的生产级 RAG，不是“切完就向量化”，而是：

> **先决定知识单元是什么，再决定检索粒度是什么，最后决定给 LLM 回填多少上下文。**

这三个层次必须分清。

---

### 相关知识点补充

- 固定长度切分：简单，但容易截断语义
- 结构切分：更自然，适合 Markdown、FAQ、技术文档
- 语义切分：效果更好，但预处理成本更高
- 父子分块：解决“检索粒度”和“生成上下文完整性”的矛盾
- 自适应切分：本质是“不同文档、不同结构、不同长度，用不同粒度策略”
- 菜单类 RAG 最适合按业务对象切分，而不是按 token 硬切

---

### 高频追问

`Q：chunk size 一般怎么选？`

**回答：**

没有绝对固定值，要结合文档类型、Embedding 模型、检索目标和下游 LLM 上下文预算来选。一般我会先保证 chunk 内部语义完整，再控制 token 长度不要太大，避免召回噪声和后续拼接成本过高。

`Q：什么时候一定要用父子分块？`

**回答：**

长文档、章节结构明显、单个子块难以支撑完整回答时，父子分块很有价值。尤其是技术文档、制度文档、长报告这类场景。

`Q：表格和 PDF 的切分为什么更难？`

**回答：**

因为它们不只是纯文本，还带结构信息。比如表格有行列关系，PDF 有版面层级。如果只按纯文本切，很容易把结构打散，导致召回后模型理解错误。

## 19. 上下文是如何管理的？项目里我会怎么讲？

如果面试官问我“你这个 Agent 项目里的上下文是怎么管理的”，我不会只回答一句“用了滑动窗口记忆”，因为在真实项目里，上下文其实至少分四层，它们解决的是四类完全不同的问题。

第一层是**身份上下文**。  
请求进来以后，用户端或管理端会先经过 JWT 拦截器，拦截器把 token 里的用户 ID 或员工 ID 解析出来，放到 `BaseContext` 的 `ThreadLocal` 里。这样 Controller、Service、AOP 自动填充都能读取当前登录身份。请求结束后，我又在 `afterCompletion` 里统一 `remove()`，避免线程池复用导致上下文串线。这一层解决的是“当前请求是谁”的问题，它属于 Web 请求上下文，不属于大模型记忆。

第二层是**会话上下文**。  
真正进入 AI 对话链路时，`AgentController` 会取出当前登录用户 ID，然后把它作为 `memoryId` 传给 `FoodRecommendationAgent.chat()`。在 `AgentConfig` 里，我通过 `chatMemoryProvider(memoryId -> MessageWindowChatMemory.withMaxMessages(10))` 给每个 `memoryId` 分配一个独立的窗口记忆。这样同一个用户前几轮说过的“不要太辣”“预算 30 左右”“一个人吃”这些约束就能保留下来，模型不会每一轮都只看当前这一句。这一层解决的是“当前会话最近几轮在说什么”的问题。

第三层是**检索上下文**。  
我的项目不是把全量菜单硬塞进 prompt，而是先做 RAG。系统启动时把在售菜品和套餐向量化，用户发起问题后，再用 `ContentRetriever` 从 `EmbeddingStore` 里召回相关菜单片段。当前配置是 `maxResults(5)`、`minScore(0.5)`，也就是最多回填 5 条、相似度低于阈值的内容不进上下文。这样做的目的是把“和当前问题最相关的菜单知识”注入模型，而不是让模型在全量菜单里自己瞎找。这一层解决的是“模型回答时应该参考哪些外部知识”的问题。

第四层是**工具上下文**。  
除了静态菜单知识，我还把历史订单、当前在售菜品、当前在售套餐这些动态业务能力封装成 `@Tool`。模型需要动态数据时，会调用 `AgentTools` 获取结果，再把这些结果作为生成依据的一部分。也就是说，工具返回结果本质上也是上下文，只不过它不是检索出来的知识片段，而是业务系统实时返回的结构化事实。这一层解决的是“模型还缺哪些实时业务信息”的问题。

所以如果让我用一句话总结，我会说：

> 我的项目里上下文不是单一概念，而是“身份上下文 + 会话上下文 + 检索上下文 + 工具上下文”四层协同。身份上下文保证请求安全隔离，会话上下文保证多轮连续性，检索上下文提供菜单知识，工具上下文补充实时业务数据，最终一起构成模型生成时的有效输入。

---

### 最终是怎么拼成一次模型输入的？

如果再往下拆，我会把一次完整调用理解成下面这个顺序：

1. 用户请求到达接口，JWT 拦截器恢复当前用户身份，写入 `BaseContext`
2. `AgentController` 从 `BaseContext` 中取出用户 ID，作为 `memoryId`
3. 请求先进入 `AgentChatService`，生成 `requestId`，并记录 `userId`、`memoryId`、输入长度等基础日志
4. 然后再调用底层 AiService，也就是 `FoodRecommendationAgent`
5. RAG 检索器从向量库中召回相关菜单片段
6. 如果模型判断需要动态数据，再触发 `@Tool` 查询历史订单、在售菜品、在售套餐
7. LangChain4j 将系统提示词、最近窗口记忆、检索结果、工具结果和当前用户输入一起组织成最终上下文
8. 模型生成答案返回，`AgentChatService` 再记录输出长度、耗时和异常信息

这里有一个面试里很好用的点：  
**系统提示词、记忆、RAG、Tool 其实都可以看成“上下文构造”的不同来源。**  
真正的工程难点不是“模型调起来没有”，而是“什么信息该进上下文、什么不该进、以什么形式进、进多少”。

---

### 为什么不能把所有信息都塞进上下文？

这是一个非常容易被追问的问题。  
很多人会下意识觉得“上下文越多越好”，但在真实系统里恰恰相反，**无控制地堆上下文通常会让效果变差**。

原因主要有四个。

第一，`token` 成本会快速上升。  
如果把全量历史对话、全量菜单、全量历史订单都塞进去，输入成本会非常高，而且推理延迟也会上升。

第二，噪声会污染模型注意力。  
用户当前只是想找“一个人吃、不辣、30 元左右”的套餐，你如果把所有历史聊天和几十条无关菜品都丢进去，模型注意力会被分散，反而更容易答偏。

第三，很多历史内容其实不是稳定偏好。  
比如用户上周说“今天想吃辣”，这不代表他长期都爱吃辣，所以历史信息不能无限累积，应该分成短期会话记忆和长期稳定画像。

第四，越多上下文不代表越可信。  
如果工具返回是非结构化大文本、检索片段过多、历史记忆过长，模型会更难判断哪些信息优先级更高。

所以我在项目里采用的是一个“分层注入、按需补充、控制上限”的思路，而不是简单地把所有数据喂给模型。

---

### 当前实现的优点和边界

如果面试官继续追问“你这个方案有没有边界”，我会主动承认有，而且这反而会显得更专业。

第一，当前项目里 `memoryId` 用的是用户 ID，这适合快速落地和单会话连续推荐。  
但如果同一个用户在多个端、多个页面同时开启不同对话，直接用用户 ID 作为记忆主键，就可能出现多个会话共享同一窗口的问题。更生产级的设计应该是 `userId + conversationId`，用会话 ID 做更细粒度隔离。

第二，当前 `MessageWindowChatMemory` 是进程内窗口记忆。  
这意味着服务重启后记忆会丢失，多实例部署时也不能天然共享。生产环境如果希望会话跨实例、跨节点连续，一般会把短期记忆外置到 Redis 或数据库。

第三，当前向量存储也是 `InMemoryEmbeddingStore`。  
这适合开发和演示，但生产里应该切到持久化向量库，比如 Chroma、Milvus、PgVector 等，不然服务重启后检索上下文也会丢。

第四，工具调用里的“身份”不能完全信模型自己传。  
当前工具方法里有 `getUserLastOrders(Long userId)` 这样的参数，从工程最佳实践上讲，生产环境应该由服务端绑定当前登录用户身份，而不是完全让模型自己决定传哪个用户 ID，否则会有越权风险。也就是说，**用户身份属于受控上下文，不能交给模型自由生成。**

第五，当前项目已经有第一版可观测性，但还不是全量细粒度拆解。  
我现在已经能做到请求级别的 `requestId`、输入输出长度、耗时、异常日志，以及缓存侧的 hit/miss 指标统计；但像 `retrieval time`、`tool time`、`token usage`、工具级错误分类这些更细粒度观测，还可以继续往下补。

### 高频追问

`Q：你项目里的上下文到底分哪几类？`

**回答：**

我会分成四类。第一类是身份上下文，也就是当前请求对应的用户或员工是谁，这部分由 JWT + ThreadLocal 维护；第二类是会话上下文，也就是当前用户最近几轮对话内容，这部分由 LangChain4j 的窗口记忆维护；第三类是检索上下文，也就是从菜单知识库里召回的相关菜品、套餐信息，这部分由 RAG 提供；第四类是工具上下文，也就是从订单服务、菜品服务、套餐服务中实时拿到的动态业务数据。  
这四类上下文解决的问题不一样：身份上下文解决权限和数据隔离，会话上下文解决多轮连续性，检索上下文解决模型缺少最新菜单知识，工具上下文解决模型拿不到实时业务事实。真正好的 Agent 系统，不是单纯“上下文越长越好”，而是把不同来源的信息按边界组织好。

`Q：为什么你要用滑动窗口记忆，而不是把所有历史聊天都带上？`

**回答：**

因为多轮对话里，真正高相关的通常是最近几轮，而不是全部历史。  
如果把所有历史都带上，第一会导致 token 成本和延迟持续上升；第二会引入大量过期信息和噪声，让模型不容易抓住当前重点；第三有些内容只是短时需求，不应该一直保留，比如用户某一轮说“今天想吃辣”，这不能直接等同于长期偏好。  
所以我在当前项目里选择了 `MessageWindowChatMemory.withMaxMessages(10)`，用固定窗口保存最近 10 条消息。这个值不是绝对标准，而是结合当前业务复杂度做的取舍。外卖推荐场景下，用户意图一般在最近几轮就能形成闭环，10 条足够覆盖“口味 + 预算 + 人数 + 套餐/单品偏好”这类约束，同时又不会把上下文拉得过长。  
如果后面要继续升级，我会把“短期窗口记忆”和“长期偏好画像”拆开。短期记忆只保留最近几轮，长期记忆则把稳定偏好抽成结构化标签，比如偏辣、预算带、常点品类，而不是保存无限历史原文。

`Q：为什么你现在用用户 ID 做 memoryId？这样会不会有问题？`

**回答：**

当前这样做的主要原因是实现简单，而且能快速让同一个登录用户在连续提问时保留上下文。对演示版和轻量业务场景来说，这个方案是够用的。  
但它确实有边界。如果同一个用户同时在多个页面、多个设备发起不同对话，直接用用户 ID 作为 memoryId，就可能把本来应该独立的两段会话混在一起。  
所以如果按生产级设计，我会把记忆主键升级成 `conversationId`，并且建立 `conversationId -> userId` 的归属关系。这样同一个用户可以有多个独立会话，每个会话都有自己的窗口记忆，既能保证上下文连续，也能避免串线。  
再往前走一步，还可以给会话设置 TTL、最近访问时间和状态位，比如进行中、已结束、已归档，这样整个上下文管理会更像一个完整的会话系统，而不只是一个内存窗口。

`Q：RAG 检索结果为什么也算上下文？你是怎么控制它不失控的？`

**回答：**

因为模型最终看到的输入，不只有用户原始问题，还包括外部知识补充。  
在我的项目里，模型本身并不知道当前真实有哪些在售菜品和套餐，所以必须先通过 RAG 把相关菜单知识检索出来，再作为上下文注入。这个过程本质上就是“给模型补上下文”。  
为了避免检索结果失控，我做了两个基本限制。第一是限制召回数量，也就是当前配置 `maxResults(5)`，不会无限召回。第二是相似度阈值控制，也就是 `minScore(0.5)`，低于阈值的片段不进入上下文，尽量减少无关噪声。  
如果面试官再深挖，我还会补充：更生产级的方案里，我会继续做查询改写、结果去重、重排、上下文压缩，甚至区分“检索粒度”和“回填粒度”。比如先用较小 chunk 做召回，再把它的父块或结构化摘要回填给模型，兼顾精准性和上下文完整性。

`Q：工具返回结果为什么也要单独管理？为什么不能把整个订单信息都塞进去？`

**回答：**

工具结果也是上下文，但它和 RAG 检索到的知识片段不同，它更偏“实时事实”。  
比如用户最近点过什么、当前有哪些在售套餐，这些信息属于业务系统实时状态，必须通过工具拿，而不是让模型猜。  
但工具结果也不能无控制扩张。比如历史订单如果把明细原文全部返回，既浪费 token，也不利于模型抓重点。所以当前项目里我在 `getUserLastOrders` 里做的是提取最近订单中的菜品名称、去重、限制数量，最多返回 10 个名字。这个设计思路本质上就是“结构化摘要优于原始大文本”。  
如果继续增强，我会把工具结果做得更结构化，比如明确区分 `top_categories`、`recent_dishes`、`price_preference` 这些字段，而不是直接给模型一大段自然语言。因为越结构化，模型越容易稳定使用，也更利于后续可观测和审计。

`Q：为什么你在工具里强调显式传 userId，而不是直接从 ThreadLocal 取？`

**回答：**

这是一个很容易体现工程意识的点。  
`ThreadLocal` 更适合 Web 请求线程里的身份恢复，比如 Controller、Service、AOP 这种同步调用链。可一旦进入 AI 工具调用、异步线程池、跨服务调用这些场景，继续依赖 ThreadLocal 就会有两个风险：第一，线程切换后上下文拿不到；第二，线程池复用或异步执行时容易把身份带错。  
所以在我的项目里，像 `getUserLastOrders(Long userId)` 这种工具方法，我更强调显式传参，而不是依赖 ThreadLocal 隐式读取。这样工具方法的输入边界更清晰，也更利于后续测试和跨线程调用。  
当然，从更严格的安全视角讲，生产环境里还要再往前走一步，不应该完全让模型自由决定 userId，而是由服务端把当前认证身份绑定到工具调用上下文中。换句话说，**上下文可以传，但身份一定要受控传递。**

`Q：ThreadLocal 上下文为什么一定要在 afterCompletion 里清理？`

**回答：**

因为请求线程通常来自线程池，线程会被复用。  
如果这次请求结束后没有清理 `ThreadLocal`，下一个请求复用同一条线程时，就可能读到上一个请求残留的用户 ID，这类问题非常隐蔽，往往不是直接报错，而是出现“偶发串数据”“A 用户看到 B 用户数据”这种严重问题。  
之所以选择在 `afterCompletion` 清理，而不是更早的位置，是因为它发生在整个请求链路完成之后，这时 Controller、Service、AOP 都已经使用完当前身份了，清理时机最稳。过早清理会导致后续链路拿不到上下文，过晚清理又会污染线程。  
所以我会把这部分总结成一句面试话术：JWT 解决的是认证，ThreadLocal 清理解决的是线程复用下的上下文安全。这两个点要分开讲。

`Q：如果上下文越来越长，你会怎么压缩？`

**回答：**

我会把压缩拆成三层。  
第一层是对话层压缩，也就是最近几轮保留原文，更早的轮次做摘要，只保留对当前决策有帮助的信息，比如口味约束、预算、人数组合。  
第二层是知识层压缩，也就是 RAG 召回不要把整段长文本都塞进去，而是做结构化抽取、关键句提炼、父子分块回填。  
第三层是工具层压缩，也就是工具返回不要给大段原始 JSON，而是转成模型真正需要的几个关键字段。  
这三个层次结合起来，本质上就是把“原始上下文”转成“决策上下文”。因为模型真正需要的不是越多越好，而是越相关越好。

`Q：如果把当前项目升级成生产级，你会怎么做更完整的上下文管理？`

**回答：**

我会重点做六件事。  
第一，把 `memoryId` 从 `userId` 升级成 `conversationId`，支持同一用户多会话隔离。  
第二，把短期记忆从进程内存迁移到 Redis 或数据库，支持多实例共享和会话恢复。  
第三，把长期偏好从对话原文里抽离出来，做成结构化用户画像，比如口味、预算、常点品类、忌口标签。  
第四，对 RAG 检索链路增加重排、摘要和缓存，控制知识上下文的质量和长度。  
第五，对工具上下文做更严格的身份绑定和权限校验，避免模型越权调用。  
第六，给整个上下文链路加可观测性，记录每次调用到底用了哪些记忆、哪些检索片段、哪些工具结果、最终 token 消耗多少，这样后面排查效果问题才有抓手。  
如果把这六件事补齐，那上下文管理就不是单一的“消息窗口”，而是一个真正可上线、可扩展、可追踪的上下文系统。

## 21. 项目治理：双写不一致、压测和慢接口排障怎么讲？

### 21.1 你项目里的双写不一致怎么理解？有没有用中间件？

如果面试官问到双写不一致，我会先把问题边界讲清楚：  
所谓双写不一致，指的是**数据库和缓存同时涉及更新时，两边状态短时间不一致**。在我这个项目里，典型场景就是菜品、套餐、店铺状态这些热点数据。

当前项目的真实做法不是“写数据库同时写缓存”，而是更标准的 **Cache Aside 模式**：

1. 读请求先查 Redis
2. Redis 没命中再查 MySQL
3. 把数据库结果回填缓存
4. 写请求先更新数据库
5. 更新成功后主动删除对应缓存

我为什么不用“先更新数据库再直接更新缓存”？  
因为缓存不是权威数据源，数据库才是。  
如果直接双写更新缓存，会有两个问题：

- 写路径更重，业务代码更复杂
- 一旦数据库成功、缓存失败，或者缓存成功、数据库失败，就更容易出现脏数据

所以我当前项目的做法是：**数据库为准，缓存做派生副本，写操作走“更新 DB + 删除缓存”**。  
你现在在代码里能看到的就是：

- 用户端热点数据统一通过 `HotDataCacheService` 读取
- 管理端修改菜品、套餐、店铺状态后，会主动失效缓存
- 再通过随机 TTL 和命中/未命中指标做治理

如果面试官继续追问：

> 那这样就绝对不会不一致了吗？

我会回答：不会。  
它只能做到**工程上足够可靠，而不是理论上的绝对强一致**。

典型不一致窗口有两个：

第一种情况：  
A 线程更新数据库成功，但删缓存前，B 线程正好把旧值读出来并回填缓存，导致缓存里短时间又出现旧值。

第二种情况：  
数据库更新成功了，但删缓存动作因为网络抖动、Redis 超时或者进程异常没执行成功，旧缓存会一直存在到 TTL 到期。

所以当前项目对这个问题的处理思路是：

- 先用 Cache Aside 降低复杂度
- 后台写操作主动失效热点缓存
- 给热点缓存设置随机 TTL，避免脏数据长期固化
- 通过缓存指标监控发现异常命中和回源波动

如果面试官问到“你没听过那个中间件”，大概率他说的是 **Canal / Debezium 这类基于 Binlog 的 CDC 中间件**。  
这个我会这样解释：

> Canal 本质上不是 Redis 中间件，而是 MySQL Binlog 订阅中间件。它会伪装成 MySQL 从库去订阅 Binlog，数据库一旦有变更，它就能拿到变更事件，然后由下游消费者决定删缓存、发消息或者同步搜索索引。

也就是说，如果未来我把当前项目从单体外卖系统扩展到更高并发、多实例部署，我会考虑两种增强方案：

第一种，延迟双删。  
也就是更新数据库后先删一次缓存，过一小段时间再删一次，尽量覆盖并发读导致的旧值回填窗口。

第二种，Binlog CDC。  
也就是数据库提交成功后，由 Canal 或 Debezium 监听 Binlog，再异步通知缓存失效。这样缓存删除动作就从业务线程里解耦出来，可靠性更强，也更适合多个下游系统一起消费变更。

所以这题最稳妥的答法是：

> 当前项目里我用的是 Cache Aside + 主动失效缓存，这是已经真实落地的方案；如果继续往高并发、多实例场景演进，我会再引入延迟双删或者 Canal 这类 Binlog CDC 中间件，把缓存一致性从“业务线程内删除缓存”升级成“数据库变更驱动缓存失效”。

### 21.2 你有做过具体压测吗？怎么做的？

如果面试官问我“你有做过具体压测吗”，我会按两层来答：**SQL 层压测** 和 **接口层压测**。

第一层是 SQL 层。  
我会先从交易主链路里挑高频 SQL，比如：

- 用户端订单分页
- 超时订单扫描
- 购物车查重
- 默认地址查询
- 菜品分类查询
- 套餐分类查询

然后基于接近真实业务分布的数据做基线压测。  
工具上我一般会用：

- `mysqlslap` 或 `sysbench` 做 SQL 级压测
- `EXPLAIN` 看执行计划
- MySQL 慢查询日志看慢 SQL

压测时我会按并发梯度跑，比如 10、50、100、200 逐步拉升，并且每轮都做预热，避免把冷启动噪声算进去。  
核心关注的指标有：

- 平均响应时间
- P95 / P99
- QPS
- 错误率
- 扫描行数
- 是否回表
- 是否命中索引

第二层是接口层。  
这一层我会重点压：

- 用户端菜品查询
- 套餐查询
- 下单接口
- 订单分页查询
- WebSocket 通知相关场景

工具上可以用 `JMeter`、`wrk` 或 `ab`。  
比如压菜品查询接口时，我会分别测：

- 无缓存时的数据库直接查询
- 加热点缓存后的查询
- 管理端更新后缓存失效再回填的过程

这样可以把“有没有缓存”和“缓存命中后收益多大”分开看。

如果面试官继续追问“你怎么判断优化真的有效”，我会说我不会只看响应时间下降，还会同时看三件事：

1. `EXPLAIN` 有没有从全表扫描收敛到索引访问
2. 慢查询日志条数有没有下降
3. 压测口径一致的前提下，P95 和错误率有没有同步改善

所以这题我会总结成一句：

> 压测不是只跑一下接口看个 QPS，而是先找到高频 SQL 和高频接口，再结合执行计划、慢查询和接口时延一起看，最后用同一口径复测，确认优化前后差异。

### 21.3 场景题：接口响应很慢，但 CPU 和内存都不高，怎么排查？有没有无侵入式手段？

这道题我会先给结论：  
**CPU 和内存都不高，但接口很慢，通常不是“算不动”，而是“在等”**。  
最常见的是在等数据库、Redis、下游 HTTP、磁盘 I/O、锁、线程池资源、连接池资源，或者文件描述符耗尽。

我一般按五个层次排：

第一层，先看是不是下游慢。  
因为业务线程很可能卡在 MySQL、Redis、第三方支付、地图服务或者模型接口上。  
这个时候应用本身 CPU 不高很正常，因为线程大多在阻塞等待。

第二层，看线程状态。  
如果大量线程都在 `WAITING`、`TIMED_WAITING`、`BLOCKED`，那说明瓶颈大概率不在 CPU，而在锁、I/O 或连接。

第三层，看连接池和文件描述符。  
比如数据库连接池满了、HTTP 连接池满了、Redis 连接耗尽了，线程会排队等待。  
再比如进程打开的 socket、文件句柄太多，接近上限，也会导致新连接建立变慢甚至失败。

第四层，看磁盘和网络。  
比如磁盘 I/O wait 高、网络 retransmission 高、下游响应时间变长，也会导致接口整体变慢，但 CPU 和内存看起来并不高。

第五层，看 GC 和锁。  
有时候不是内存占用高，而是 Young GC 太频繁、或者某个锁争用严重，也会把响应时间拉长。

如果面试官问“有没有无侵入式的方式”，我会强调 **先用系统命令和诊断工具，不要第一时间改代码或打断点**。

我在线上 Linux 环境通常会这样做：

1. 先看进程和线程概况  
`top -Hp <pid>`  
看是不是某些线程一直阻塞，或者某一类线程池打满。

2. 看线程维度 CPU 和等待情况  
`pidstat -p <pid> -t 1`

3. 看 JVM 线程栈  
`jstack <pid>`  
或者  
`jcmd <pid> Thread.print`

这里重点看线程是不是卡在：

- `java.net.SocketInputStream`
- `com.mysql`
- `redis`
- `wait`
- `lock`

4. 看 GC  
`jstat -gcutil <pid> 1000`

5. 看磁盘 I/O  
`iostat -x 1`

6. 看系统整体等待  
`vmstat 1`

7. 看网络连接  
`ss -s`  
`ss -antp | grep <pid>`

8. 看文件描述符  
`lsof -p <pid> | wc -l`  
或者  
`ls /proc/<pid>/fd | wc -l`

再结合：

`cat /proc/<pid>/limits | grep "open files"`

看当前 FD 是否接近上限。

如果面试官提到“看 fd 文件描述符”，你就可以这么解释：

> 很多服务明明 CPU 和内存都不高，但如果数据库连接、HTTP 连接、日志文件、socket 句柄太多，接近 `open files` 限额，新的连接创建就会异常慢。这时候就要看 `/proc/<pid>/fd` 目录数量或者 `lsof -p <pid>` 的输出，再结合进程限制去判断是不是句柄泄漏或者连接没及时释放。

除此之外，我还会配合中间件侧一起看：

- MySQL：`show processlist`、慢查询日志
- Redis：`redis-cli --latency`、`info stats`
- Nginx：`upstream_response_time`

如果想再进一步，但依然不改代码，我会用 **Arthas**。  
比如：

- `thread` 看线程
- `trace` 跟方法耗时
- `watch` 看参数和返回值
- `dashboard` 看 JVM 状态

所以这道题我最后会总结成：

> CPU 和内存都不高但接口慢，优先考虑阻塞等待而不是计算瓶颈。排查顺序一般是线程栈、连接池、下游依赖、I/O、FD、锁和 GC；无侵入式手段优先用 `top/pidstat/jstack/jstat/iostat/ss/lsof/Arthas`，而不是先改代码。

## 22. 为什么在研发过程中想到引入 AI？底层用了哪个大模型，为什么选它？

这个问题我一般会分成“业务出发点”和“模型选型”两部分回答。

先说业务出发点。  
我这个项目原本是一个比较完整的 B/C 双端外卖交易系统，已经有商品、套餐、购物车、下单、支付、订单流转这些标准能力。  
但我在看用户端交互的时候发现一个问题：传统点餐系统本质上还是**关键词检索 + 用户自己筛选**。  
也就是说，系统只能回答“菜单里有什么”，但很难回答“我今天不想吃太辣、预算 30 左右、一个人吃，你直接推荐点什么”。

所以我引入 AI 的出发点，不是为了做一个聊天功能，而是为了把用户交互从“被动检索”升级成“基于意图理解的主动推荐”。  
这件事只有把大模型和业务系统结合起来才有意义，因为它不只是文本生成，还涉及：

- 理解口味、预算、人数组合
- 检索当前真实在售菜单
- 结合历史订单偏好
- 必要时调用业务工具
- 最后输出能直接点单的建议

所以我会明确告诉面试官：

> 我引入 AI 不是为了做一个聊天 demo，而是为了补传统交易系统在“意图理解”和“推荐生成”上的短板。

再说底层模型。  
如果严格按照当前项目代码来讲，我现在接的是 `gpt-3.5-turbo`，Embedding 用的是 `all-MiniLM-L6-v2`。  
我选它的原因主要有四个：

第一，LangChain4j 集成成本低。  
我当前项目是 Java 技术栈，LangChain4j 对 OpenAI 兼容模型的接入比较顺，能快速把 AiService、RAG、Tool Calling、多轮记忆这几层跑通。

第二，成本和效果比较平衡。  
这个场景不是极重推理场景，更强调对菜单意图的理解、工具调用稳定性和多轮对话连续性，`gpt-3.5-turbo` 在这个阶段已经够用。

第三，工具调用和多轮能力成熟。  
对我这个项目来说，模型不是只回答一句话，而是要配合 RAG、工具和记忆去做组合决策，所以模型的“工程可集成性”比单纯 benchmark 更重要。

第四，方便后续替换。  
我当前是通过 LangChain4j 的抽象接入模型，所以后续如果换成中文效果更好的模型，比如 Qwen、DeepSeek 这一类兼容 OpenAI 协议的模型，迁移成本也不会太高。

所以这道题我最后会收成一句话：

> 引入 AI 的出发点是把传统点餐系统从关键词检索升级成意图驱动推荐；当前模型选型更重视 Java 生态集成、成本、工具调用和多轮能力，而不是单纯追求模型越大越好。

## 23. 知识库问答 / RAG 流程怎么设计？每一部分分别做什么？

我会把这题答成一条完整链路，而不是散点概念。

如果对应我当前项目，整个知识库问答链路可以拆成六步：

第一步，知识准备。  
我当前项目的知识载体不是长文档，而是菜单数据，所以我会在系统启动时把**起售菜品和套餐**从数据库里加载出来。

第二步，知识组织。  
我不会把原始表字段直接丢给模型，而是先把每个菜品或套餐组织成适合检索的文本，比如：

- 菜名 / 套餐名
- 分类
- 价格
- 描述

这一步的本质是把结构化业务对象转成适合语义检索的知识单元。

第三步，向量化与入库。  
把上一步的文本做 Embedding，然后写入 `EmbeddingStore`。  
我当前项目里因为是演示和轻量场景，用的是 `InMemoryEmbeddingStore`；如果上生产会换成持久化向量库。

第四步，查询理解与召回。  
用户提问后，先把 Query 做 Embedding，再到向量存储里召回最相关的菜单知识。  
当前项目里我对召回做了两个基本限制：

- `maxResults(5)`
- `minScore(0.5)`

也就是最多取 5 条，相似度太低的结果不放进上下文。

第五步，工具补充动态信息。  
如果只有静态菜单知识还不够，比如需要用户历史偏好、当前在售菜单、当前在售套餐这些动态业务事实，我就通过 `@Tool` 去查业务服务。

第六步，统一构造上下文交给模型生成。  
也就是把：

- 系统提示词
- 检索结果
- 工具结果
- 多轮记忆
- 当前用户输入

一起交给模型，让它输出最终回答。

所以如果面试官问“知识库问答分为哪几个部分”，我会总结成：

> 知识准备、知识组织、向量化存储、查询召回、工具补充、最终生成，这六步缺一不可。RAG 的价值不是“先搜再答”这么简单，而是把外部知识和模型生成统一编排成一个稳定链路。

## 24. 如何评估和优化 RAG？BM25、查询改写、Prompt 构建、下一步优化怎么答？

这块我一般会按“评估、优化、原理、落地”四层来答。

先说评估。  
RAG 不能只看最终回答好不好看，而要先拆成**检索效果**和**生成效果**。

检索层我会重点看：

- Recall@K
- Hit Rate
- MRR
- 召回结果里是否包含真正答案依据

生成层我会看：

- 回答是否引用了正确菜单
- 有没有推荐菜单里不存在的菜
- 理由是否和召回内容一致
- 用户问题约束有没有被满足

如果放到当前项目，我会自己构造一批问答样本，比如：

- “不辣、一个人吃、30 元左右吃什么”
- “有没有清淡点的套餐”
- “我之前常点的菜里类似的还有什么”

然后人工标记每个问题理想应该召回哪些菜单项，再看 topK 有没有把关键菜单召回来。

再说怎么优化召回。  
我一般会从六个方向做：

第一，优化知识切分。  
我当前项目里本质上是按业务对象切分，每个菜品或套餐就是一个 chunk，这比固定 token 切分更适合菜单场景。

第二，优化文本组织。  
不是只放菜名，而是把菜名、分类、价格、描述一起组织进去，增强语义表达。

第三，做查询改写。  
比如用户说“来点一个人吃的，不辣，别太贵”，我会把它抽成更清晰的检索意图：

- 用餐人数 = 1
- 口味 = 不辣 / 清淡
- 预算 = 低价位
- 类型 = 菜品或套餐

当前项目里还没有单独做查询改写模块，但如果面试官问，我会说：  
我会优先做**轻量规则改写 + LLM 辅助改写**。  
规则负责把预算、人数、辣度这种明确约束提出来；LLM 负责同义表达归一化和补全检索语义。

第四，做混合检索。  
向量检索适合语义理解，但像菜名、品牌词、精确实体，BM25 这类词法检索往往更稳。  
所以更完整的方案一般会做 dense + sparse 混合召回。

第五，重排。  
如果 topK 结果里有噪声，可以再做 rerank，把最符合当前问题约束的结果排前面。

第六，控制上下文注入。  
召回不是越多越好，要控制数量、相似度阈值、去重和压缩，避免无关片段把模型带偏。

再说 BM25 是怎么检索的。  
我会这样答：

> BM25 本质上是基于词项频率、逆文档频率和文档长度归一化做打分的稀疏检索算法。  
> 它会关心某个词在当前文档里出现得多不多、这个词在全库里是不是稀有词，以及文档本身是不是太长。  
> 所以它非常适合菜名、品牌名、口味词、套餐名这类实体检索。比如用户直接问“宫保鸡丁”“酸菜鱼套餐”，BM25 往往会比纯向量检索更稳定。

再说最终 Prompt 怎么构建。  
我通常会把输入分成五块：

1. System Prompt：定义角色和边界  
2. 用户原始问题  
3. 检索召回的菜单上下文  
4. 工具返回的动态业务数据  
5. 最近几轮对话记忆

其中 System Prompt 我会明确约束：

- 你是点餐助手
- 不要推荐菜单中不存在的菜
- 优先结合用户约束和在售菜单回答
- 如果信息不足，先澄清再推荐

这一步非常关键，因为 Prompt 不是拿来“教模型全世界知识”的，而是拿来**约束模型怎么使用检索结果和工具结果**。

最后说下一步优化。  
如果继续增强当前项目，我会优先补：

- 持久化向量库，避免重启丢数据
- 混合检索，加入 BM25
- 查询改写模块
- 重排模型
- 会话摘要记忆
- 更细粒度 Agent 观测
- 检索质量评估集

所以这一题最后我会总结成：

> RAG 效果提升不是只换一个模型，而是从知识组织、查询改写、召回策略、重排、Prompt 构建和观测评估一整条链路去优化。

## 25. Redis 在项目里怎么用？数据类型、底层结构、应用场景和“Redis 线程池”怎么答？

如果面试官从 Redis 往下追，我会分成“项目怎么用”“常见数据类型”“底层实现”“线程模型”四部分讲。

先说项目怎么用。  
在我当前项目里，Redis 主要承接的是热点缓存，而不是复杂分布式协调。  
最典型的几个 key 是：

- 菜品分类缓存
- 套餐分类缓存
- 店铺状态缓存

也就是说，我当前项目里 Redis 最主要用的是 **String 类型**，值里存的是经过 JSON 序列化后的对象或列表。  
这样做的优点是：

- 读写简单
- 和 Spring Data Redis 集成方便
- 对菜单、套餐、店铺状态这类整体读取的数据结构很合适

但如果面试官问“了解哪些 Redis 数据类型”，我不会只说 String，我会完整回答：

- String：缓存对象、计数器、分布式锁
- Hash：对象属性拆分存储
- List：消息队列、时间线
- Set：去重、共同好友、标签
- ZSet：排行榜、延时队列、带分数排序
- Bitmap：签到、布尔状态压缩
- HyperLogLog：UV 估算
- Stream：消息流

再说底层实现。  
我一般不会把 Redis 讲成“一个大 Map”，而是会强调不同数据结构底层实现不同：

- String 底层是 `SDS`
- Hash / Set / ZSet 底层会根据数据规模在紧凑结构和哈希结构之间切换
- ZSet 常见是 `skiplist + dict`
- List 新版本偏向 `quicklist`

也就是说，Redis 的快不是单一原因，而是：

- 内存操作
- 单线程事件循环避免锁竞争
- 高效数据结构
- I/O 多路复用

再说“Redis 线程池用的哪个”。  
这个问题很多人会被问懵，因为**业务里通常不会说“我用了 Redis 线程池”**。  
我在当前项目里更准确的答法是：

> 我项目里没有单独做一个“Redis 线程池”。Redis 访问本身是通过 Spring Data Redis 完成的，客户端层通常是 Lettuce，它底层用的是 Netty 事件循环和连接管理；而我项目里真正显式配置的线程池，是 WebSocket 异步通知线程池，不是 Redis 线程池。

如果面试官继续追问，我会再补一句：

> 如果是问 Redis 客户端连接管理，Jedis 更强调连接池；Lettuce 默认连接是线程安全、可复用的，底层依赖 Netty 的 I/O 线程模型。

所以这一题你最稳的答法是：

> 当前项目里 Redis 主要用于热点缓存，数据类型以 String 为主；底层我知道 Redis 的核心不是“一个线程处理一切”那么简单，而是事件循环 + 高效内存数据结构；至于所谓 Redis 线程池，项目里没有单独配这个概念，真正配置的是业务异步线程池，Redis 客户端侧一般由 Lettuce/Jedis 自己做连接和 I/O 管理。

## 26. Redis 的 TTL、过期删除和淘汰策略怎么讲？不同场景怎么选？

这题我会先把两个概念分开：

- **过期删除**：某个 key 自己到了 TTL
- **内存淘汰**：Redis 内存不足时主动踢 key

先说 TTL 是怎么实现的。  
Redis 给 key 设置过期时间后，不是到点立刻开一个线程去删，而是主要靠两种机制：

第一，惰性删除。  
当某个 key 被访问时，Redis 会先判断它有没有过期，过期了就直接删掉，不返回旧值。

第二，定期删除。  
Redis 会周期性抽样检查带 TTL 的 key，把已经过期的删掉。

这两个机制配合起来，平衡了 CPU 开销和内存回收效率。

再说内存淘汰策略。  
如果 Redis 设置了最大内存，就会有淘汰策略，比如：

- `noeviction`
- `volatile-lru`
- `allkeys-lru`
- `volatile-lfu`
- `allkeys-lfu`
- `volatile-random`
- `allkeys-random`
- `volatile-ttl`

我一般会这样理解：

- `volatile-*` 只在设置了 TTL 的 key 里选
- `allkeys-*` 在全量 key 里选
- `lru` 关注最近最少使用
- `lfu` 关注访问频率
- `ttl` 优先淘汰更快过期的 key

不同场景选择也不一样。

如果 Redis 主要就是缓存，而且允许数据被淘汰，我更倾向：

- `allkeys-lru`
- 或 `allkeys-lfu`

如果热点访问分布非常明显，我更喜欢 `allkeys-lfu`，因为它比 LRU 更能识别长期热点。

如果 Redis 不只是缓存，还放了部分不能随便丢的数据，那我就不建议让它混部，或者至少把这类数据拆到单独实例。  
因为一旦用了 `allkeys-*`，就意味着所有 key 都有被淘汰风险。

如果面试官问“你项目里为什么用了 TTL 和随机过期”，我会说：

> 因为我的项目里菜品、套餐、店铺状态这些数据属于典型热点缓存。TTL 是为了避免缓存永久不刷新，随机过期是为了避免同一批 key 同时失效造成瞬时回源高峰。

所以这题我最后会总结成一句：

> 过期删除解决的是“key 到时间该不该删”，淘汰策略解决的是“内存不够时先踢谁”；缓存型场景一般优先选 `allkeys-lru` 或 `allkeys-lfu`，而且最好不要把缓存和强依赖数据混在一个实例里。

## 27. Redis 如何实现高可用？主从、哨兵、Cluster 和分片原理怎么讲？

这道题我一般按“三层演进”来答。

第一层是主从复制。  
主节点负责写，从节点同步主节点数据，可以承接读流量，也可以做故障恢复基础。  
它解决的是“副本”和“读扩展”，但本身不解决自动故障转移。

第二层是哨兵。  
哨兵本质上是监控 + 自动切换机制。  
它会持续监控主从节点状态，一旦发现主节点不可用，会经过投票选举，把某个从节点提升成新的主节点。  
所以哨兵解决的是**自动主备切换**，但数据本身还是一主多从，没有真正分片。

第三层是 Cluster。  
Cluster 不只是高可用，还解决了**水平扩容和数据分片**。  
它会把 key 空间划分成 16384 个 hash slot，不同节点负责不同 slot。  
写请求会根据 key 计算 slot，然后路由到对应节点。

所以三者区别可以这么记：

- 主从：解决副本和读扩展
- 哨兵：在主从基础上解决自动故障转移
- Cluster：在高可用基础上再解决分片和横向扩容

再说为什么要分片。  
因为单个 Redis 实例再快，内存和网络能力也是有限的。  
当数据量、QPS、热点 key 数量继续上升时，单节点会变成瓶颈。  
分片的本质就是把数据和流量拆开，让不同节点共同承载。

如果面试官问“数据分片原理是什么”，我会说：

> Redis Cluster 用的是哈希槽分片。客户端对 key 做哈希运算后得到 slot，slot 再映射到具体节点。这样迁移时不是一条条 key 搬，而是迁移 slot，管理成本更低。

如果结合当前项目来讲，我会明确说：

> 我当前项目还是单体外卖系统，Redis 主要用在热点缓存，还没真正上 Redis Cluster；但如果后面热点数据量和访问量继续扩大，比如增加营销活动、更多商户数据、更多统计维度，我会优先考虑读写分离和 Cluster 扩展。

## 28. HTTP、HTTPS、TLS 握手和 QUIC 怎么讲？

这题我一般会把它答成“安全性 + 连接模型 + 性能演进”三层。

先说 HTTP 和 HTTPS 的区别。  
最核心的区别不是端口号，而是：

- HTTP 明文传输
- HTTPS = HTTP + TLS

也就是说，HTTPS 多了一层 TLS，提供：

- 身份认证
- 数据加密
- 完整性校验

所以在业务系统里，像登录、支付、订单、用户信息、AI 对话输入输出这类链路，基本都应该走 HTTPS。

再说 HTTPS 交互过程。  
如果按 TLS 1.2/1.3 的思路讲，我会说：

1. 客户端发起 `ClientHello`，告诉服务端自己支持哪些加密套件、TLS 版本等
2. 服务端返回 `ServerHello`，选择一套双方都支持的加密方案，并把证书发给客户端
3. 客户端验证证书是否合法，是否由受信任 CA 签发，域名是否匹配
4. 验证通过后，双方协商出会话密钥
5. 后续 HTTP 数据都用会话密钥加密传输

如果面试官追问“为什么 HTTPS 比 HTTP 慢”，我会说：

- 多了一次握手
- 多了加解密开销

但现代 TLS 1.3、会话复用、硬件优化之后，这个成本已经比早期小很多了。

再说 QUIC。  
我听说过，而且会这样答：

> QUIC 是基于 UDP 构建的传输层协议，HTTP/3 就跑在 QUIC 之上。  
> 它的核心目标是减少握手延迟、避免 TCP 队头阻塞，并把连接迁移做得更自然。

和 TCP + TLS + HTTP/2 相比，QUIC 有几个关键点：

- 基于 UDP，自定义可靠传输
- 传输层和加密层一体化，握手更快
- 多路复用时，一个流丢包不一定拖死所有流
- 连接 ID 机制更适合移动网络切换

如果结合业务系统，我会说：

> 当前我的外卖项目还是标准 HTTP/HTTPS 体系，没有上 HTTP/3 或 QUIC；但从原理上我知道 QUIC 解决的是低时延和弱网下的传输体验问题，尤其适合移动端和高交互场景。

所以这题最后我会总结成：

> HTTP 和 HTTPS 的核心差异是有没有 TLS；HTTPS 的价值在认证、加密和完整性；QUIC 则是在更底层的传输协议上继续优化握手时延和多路复用体验。

## 29. Java 基础：装箱拆箱、深拷贝和“方法里改对象，外面会不会变”怎么答？

先说装箱拆箱。  
装箱就是把基本类型转成包装类型，比如 `int -> Integer`；拆箱就是反过来。  
Java 里很多时候会自动发生，比如：

- 集合只能放对象
- 比较、运算时会自动拆箱

这里面最容易出问题的是两类：

第一，`Integer` 缓存池。  
比如 `Integer` 在 `-128~127` 范围内会复用对象，所以 `==` 比较有时候看起来对，有时候又不对。  
所以包装类型比较值，一般我会用 `equals`。

第二，空指针。  
比如 `Integer a = null; int b = a;`  
这里自动拆箱就会抛 `NullPointerException`。

再说深拷贝。  
我会先区分浅拷贝和深拷贝：

- 浅拷贝只复制对象本身，内部引用还是共享
- 深拷贝要把引用对象也一起复制出来

常见深拷贝方式有：

- 手写递归 copy
- 拷贝构造函数
- 序列化 / 反序列化
- JSON 序列化转一次
- MapStruct / BeanUtils 但要注意它们多数偏浅拷贝

如果在业务项目里，我更偏向：

- DTO / VO 用显式转换
- 复杂对象用手写 copy 或构造函数

因为这样可控，而且不会莫名共享内部引用。

再说“方法外面有个数据，传到方法里，方法里改了，外面会不会变”。  
这题我会很明确地回答：

> Java 永远是值传递。  
> 但如果传的是对象引用的副本，方法里通过这个引用去改对象内部状态，外面的对象当然会变；如果只是让这个引用指向新对象，外面不会变。

举个例子：

```java
void change(User user) {
    user.setName("A");
}
```

外面会变，因为改的是同一个对象内部状态。

```java
void change(User user) {
    user = new User();
}
```

外面不会变，因为这里只是让方法里的局部引用重新指向了新对象。

所以这题的关键不是“对象传引用”这种不严谨说法，而是：

> Java 是值传递，只不过对象场景下传的是引用的副本。

## 30. Spring Boot / Spring MVC / MyBatis 高频基础怎么答？

这几个问题我一般合并答，因为面试官常常是一套连问。

先说 Spring Boot 常见注解。  
如果只说 `@SpringBootApplication`、`@Controller`、`@RestController` 是不够的，我会顺手补成一组：

- `@SpringBootApplication`
- `@Component` / `@Service` / `@Repository` / `@Controller`
- `@RestController`
- `@Autowired`
- `@Configuration`
- `@Bean`
- `@Value`
- `@Transactional`
- `@RequestMapping`
- `@GetMapping` / `@PostMapping` / `@PutMapping` / `@DeleteMapping`

再说 `@Controller` 和 `@RestController` 的区别。  
我会这样答：

> `@RestController = @Controller + @ResponseBody`。  
> `@Controller` 更适合返回页面视图；`@RestController` 更适合直接返回 JSON 数据。

再说 Controller 参数传递。  
常见的几种是：

- `@PathVariable`
- `@RequestParam`
- `@RequestBody`
- `@RequestHeader`
- `@CookieValue`
- 直接接收表单对象 / Query 对象

如果结合当前项目代码，我会直接举例：

- `/{id}` 这种路径变量用 `@PathVariable`
- 分页和查询条件常见用 `@RequestParam` 或直接对象接收
- 新增 / 修改接口常见用 `@RequestBody`

再说 MyBatis 绑定值怎么做。  
这个问题面试官大概率在问 `#{}` 和 `${}` 的区别。

我会这样答：

> `#{}` 是预编译占位符，会走 `PreparedStatement`，更安全，能防 SQL 注入；`${}` 是字符串直接拼接，通常只适合表名、排序字段这类不能用预编译占位的场景，但必须严格校验白名单。

在实际项目里：

- 普通查询条件我一律优先用 `#{}`  
- 动态 SQL 用 `<if>`、`<where>`、`<set>` 这些标签配合  
- 多参数时可以用 `@Param`

所以这一组题最稳的答法是：

> Spring Boot 这块要把注解、Controller 返回值和参数接收讲清楚；MyBatis 这块核心是 `#{}` 预编译绑定、`${}` 字符串拼接的区别，以及动态 SQL 怎么写安全、怎么避免注入。

## 31. 扩展场景题：超卖、一人一单、Lua 为什么不直接写 Java？消息队列怎么保证不丢？

这组问题我会先主动说明：

> 当前外卖项目本身没有做秒杀券或抢购活动，所以仓库里没有真正的 Lua 抢券脚本和 MQ 下单链路；但如果面试官把题扩展到高并发营销场景，我会这样回答。

先说超卖和一人一单。  
如果是优惠券、秒杀券或者高并发库存扣减场景，我不会直接在 Java 里写：

1. 查库存
2. 查用户是否下过单
3. 扣库存
4. 写订单

原因是这几步如果拆成多次 Redis / DB 操作，在高并发下会有竞态条件。  
比如两个线程同时查到库存还有 1，同时都判断“这个用户还没买过”，然后一起往下执行，就容易超卖或者重复下单。

所以我会优先把“库存校验 + 一人一单校验 + 扣减资格”收敛到 **Lua 脚本** 里执行。  
因为 Lua 在 Redis 里执行时具备两个关键特点：

- 单线程执行
- 整个脚本原子完成

这就意味着：

- 不会出现检查完库存后，另一个线程又抢先扣掉库存
- 不会出现检查一人一单和扣库存之间被别人插队

为什么不直接在 Java 里写？  
因为 Java 里就算你把逻辑写在一个方法里，只要底层还是多次远程 Redis 调用，它对 Redis 来说就不是原子的，中间总会有并发窗口。

所以这一题我会这样收：

> Java 方法级原子，不等于 Redis 操作级原子；Lua 的价值就在于把多条 Redis 操作收进一次原子执行，降低网络往返，也避免并发竞态。

当然，只靠 Redis 还不够，我还会再加一层兜底：

- 数据库唯一索引
- 订单表幂等校验
- 异步落库时的重复消费防护

再说 MQ 怎么保证消息不丢失。  
我一般会分成发送端、Broker、消费端三层答。

发送端：

- 生产者确认机制
- 发送失败重试
- 本地消息表 / 事务消息

Broker：

- 持久化刷盘
- 多副本复制

消费端：

- 手动 ACK
- 消费失败重试
- 幂等消费

因为“消息不丢失”不能只靠 MQ 本身，真正可靠的是**生产端可确认、Broker 可持久化、消费端可重试且幂等**。

## 32. 订单未支付过期怎么处理？为什么当前用 Spring Task？分布式下会有什么问题？

这道题我会先严格对应当前项目回答。  
当前项目里，订单未支付过期不是靠 Redis TTL，也不是靠延迟队列，而是靠 **Spring Task 定时扫描订单表**。

具体逻辑是：

- 每分钟扫描一次
- 查询状态为 `PENDING_PAYMENT`
- 且 `order_time < now - 15min` 的订单
- 批量更新为取消状态

我为什么当前用这个方案？  
因为对这个项目来说，它有几个优点：

第一，实现简单。  
单体项目里直接基于数据库和定时任务就能跑通，不需要额外引入 MQ 或时间轮。

第二，数据库是权威状态。  
订单状态最终还是落在数据库里，所以扫描数据库能保证逻辑口径统一。

第三，便于和现有订单状态机对齐。  
比如超时取消、配送超时自动完成，这些本来就是订单状态流转的一部分。

如果面试官问“那 TTL 怎么实现”，我会回答：

> 当前项目里不是用 Redis TTL 来驱动订单过期，而是用数据库状态 + Spring Task 扫描。因为订单属于强业务数据，我更希望数据库是唯一事实来源；Redis TTL 更适合缓存，不适合直接作为订单状态流转的权威依据。

如果继续追问“分布式下会有什么问题”，我会说有三个核心问题：

第一，重复执行。  
多个实例上的定时任务同时跑，可能同一批订单被扫描多次。

第二，状态竞争。  
一个实例在取消订单，另一个实例可能正在支付回调或做别的状态变更。

第三，扫描压力。  
如果订单量继续增长，所有实例都全表扫描会放大数据库压力。

所以分布式下我会这样增强：

1. 任务层做单点调度  
比如：
- XXL-Job
- Quartz
- ElasticJob
- ShedLock
- Redis / DB 分布式锁

2. 数据层做幂等更新  
比如更新 SQL 带条件：

```sql
update orders
set status = 6
where id = ? and status = 1
```

这样就算多个实例同时执行，也只有一个能真正改成功。

3. 更高阶时改成延迟消息  
如果未来订单量更大、实时性要求更高，我会考虑：

- RocketMQ 延迟消息
- Redis ZSet 延时队列
- 时间轮 / 专门调度系统

所以这题我最后会总结成：

> 当前项目里用 Spring Task 扫描订单表，是一个适合单体项目、实现简单、和数据库状态机一致的方案；但分布式下会面临重复执行、状态竞争和扫描压力，需要靠分布式调度 + 幂等更新，或者进一步演进到延迟消息方案。
